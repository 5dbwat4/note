之前我们已经给大家介绍了C++面向对象特性里的继承机制，也详细讲解了虚函数的底层工作原理。大家之前看到，C++中虚函数的实现逻辑是：每一个包含虚函数的类，会额外生成一个全局的虚表vtable，这个表里存储的是该类所有虚函数的入口地址；而这个类的每一个对象内部，都会额外嵌入一个指向虚表的指针vpointer，也就是虚指针。

你看基类Shape，它拥有一个纯虚函数draw，所以它的虚表里对应draw的位置存放的是0，也就是空指针，代表这个纯虚函数没有具体实现。之后Ellipse类公开继承自Shape，它重写了draw函数，那么Ellipse自己的虚表里，draw的位置就会填入Ellipse版本draw函数的实际入口地址。再往后Circle类继承自Ellipse，如果它也重写了属于自己的draw方法，那么Circle的虚表里，draw对应的位置就会被替换成Circle::draw的函数地址。

基于这个机制，我们用基类指针来操作派生类对象的时候，就能实现运行时的多态。比如我们声明`Ellipse *e = new Ellipse; Circle *c = new Circle;`，调用`e->draw()`的时候，代码会顺着e的虚指针找到Ellipse的虚表，取出Ellipse::draw的地址完成调用；调用`c->draw()`就会自动调用Circle自己的draw实现。这里的内存布局保证了向上转型的绝对安全：Circle对象的内存最开头完整保留了Ellipse的子对象，而Ellipse子对象开头又完整保留了Shape的子对象，所以你把Circle指针向上转型为Ellipse指针、再转型为Shape指针的时候，根本不需要做任何地址偏移调整，就能正确找到对应的虚指针，不会出现任何访问错误。

接下来我们要讲解C++面向对象里另一个核心特性：运算符重载。运算符重载的作用，就是允许你自定义的类类型，可以像C++内置的int、double这些基础类型一样，直接用常规运算符完成操作。它本质上是另一种包装函数调用的形式，让你不需要每次都写`add(a,b)`这样的命名函数，直接写`a + b`就能完成相同的逻辑，代码的表达会更自然，可读性大幅提升。

绝大多数一元运算符和二元运算符都支持重载，比如算术运算符+、-、*、/、%，位运算符&、|、~、^，移位运算符<<、>>，比较运算符<、<=、>、>=、==、!=，还有赋值运算符=，下标运算符[]，函数调用运算符()，箭头访问运算符->，甚至new、delete以及它们的数组版本new[]、delete[]都可以自定义重载。但有几个运算符是C++语法明确禁止你重载的，分别是成员访问点号`.`、作用域解析符`::`、三元条件运算符`?:`、sizeof、typeid，以及四个转型运算符`static_cast`、`dynamic_cast`、`const_cast`、`reinterpret_cast`，这些永远不能被重载。

运算符重载有几个明确的限制：第一，你只能重载C++已经存在的运算符，不能自己凭空发明新的运算符符号来使用。第二，你不能改变运算符原本的优先级，比如加法的优先级永远低于乘法，重载之后这个规则不会有任何变化。第三，你也不能随意改变运算符的操作数个数，比如普通的二元+永远要接收两个操作数，你不能把它改造成一元运算符，当然C++原生支持的一元正号、一元负号是可以正常重载的。还有一条强制规则：重载后的运算符，必须至少有一个操作数是用户自定义的类类型，你绝对不能试图修改内置int类型的加法行为，不然会彻底打乱C++内置类型的运算规则。最后也是最重要的设计原则：重载后的运算符行为，一定要符合开发者对这个运算符的常规预期，不要把+重载成减法、把<<重载成计算乘法，这种反直觉的重载会彻底破坏代码的可读性，业内称这个原则为“最小意外原则”，也就是Principle of least surprise。

说到底C++的运算符重载根本没有什么魔法，它本质上就只是一个普通函数，唯一的特殊点就是函数名是`operator`后面紧跟对应的运算符符号。比如你重载加法运算符，对应的函数名就是`operator+`，当你写下`a + b`的时候，编译器会直接把这行代码翻译成调用`operator+(a, b)`，和你调用普通命名函数没有任何本质区别。

重载运算符有两种主流实现方式：第一种，把运算符作为类的成员函数。成员函数天生拥有一个隐式的第一个参数，也就是指向当前对象的this指针。所以如果你把二元+重载为成员函数，那么隐式的this指针自动绑定左边的操作数，你只需要显式定义一个参数接收右边的操作数就可以了。而如果你重载的是一元运算符，作为成员函数的话完全不需要额外的显式参数，唯一的操作数就是对象本身，通过this指针访问就行。把运算符实现为成员函数的最大好处，就是这个函数天然拥有访问类内部所有私有成员、保护成员的权限，可以直接操作类的全部数据，不需要额外把运算符声明为类的友元。但是它有一个明显的限制：作为隐式第一个参数的this指针，永远是当前类的类型，编译器不会对左边的操作数执行任何隐式类型转换。比如你写了一个自定义的Complex复数类，把+重载为成员函数，那么`Complex(1) + 2`是完全合法的，但是反过来写`2 + Complex(1)`就会编译报错，因为左边的操作数是int类型，没有办法直接绑定到Complex类型的this指针上。

第二种重载方式，是把运算符实现为全局的普通函数，通常需要把它声明为类的友元，这样才能访问类的私有成员。全局的`operator+`需要显式接收两个参数，分别对应加法的左右两个操作数，这时候编译器就可以对两个操作数都执行隐式类型转换，刚才说的`2 + Complex(1)`就可以正常编译运行了。

接下来我们打开Visual Studio，实际手写一个示例代码来演示运算符重载的用法。我这里自定义了一个MyArray类，内部维护一个动态分配的int类型数组，首先重载下标运算符`operator[]`，让我们自己写的数组类可以像原生数组一样直接通过`arr[i]`的形式访问元素，我把返回值定义成`int&`，这样我们不仅可以读取元素，还可以直接写`arr[i] = 100`对数组元素赋值。然后我们再把加法运算符重载为全局友元函数，实现两个MyArray对象对应位置的元素相加，返回一个全新的MyArray对象。现在我们编译运行这段代码，声明a1和a2两个自定义数组，依次赋值之后，直接写`a3 = a1 + a2`就可以完成两个数组的相加，完全和内置类型的写法一致，我们运行程序可以看到，输出的每一个元素的求和结果都是完全正确的。

这就是运算符重载最基础的用法，大家课后可以自己动手尝试，给复数类重载加减乘除运算符，还有等于比较运算符，写出来的代码会异常简洁自然，比手动调用成员函数的体验要好得多。

大家好，我们继续来讲C++面向对象程序设计里运算符重载的相关内容。

首先我们之前看到，对于像`+=`、`-=`这些二元运算符，把它做成成员函数的时候，只需要一个参数就够了，因为它的第一个操作数本身就是当前对象this指向的实例。但是像`++`这种一元运算符，做成成员函数就完全不需要参数了，对吧？这时候调用它的时候，直接写对象名`++`就可以，编译器自动把当前对象作为调用的主体。那这里要注意，有一些运算符我们最好不要定义成成员函数，要做成全局的普通函数，这部分我们接下来详细说。

那接下来讲怎么实现运算符的重载：第一种方式就是把它作为全局函数来实现，这时候你要显式写出它的第一个参数。但是这里有个问题，如果这个全局运算符函数需要访问类里面的私有成员，你直接访问是访问不到的，这时候就需要把这个全局函数声明为当前类的友元，这样它就可以正常访问类里面的私有数据了。尤其当运算符的操作是对两个参数都进行操作的时候，你把它实现为全局友元函数就非常方便。

比如说我们来举个全局友元运算符的例子，之前的String类，我们想实现加法运算符重载，用来做字符串拼接，那我们可以声明`friend String operator+ (const String& a, const String& b)`，这样这个全局函数就可以直接拿到两个String对象里的私有字符数组指针，不需要额外提供对外的公开接口去访问内部数据了。这时候你就不用为了让全局函数拿到私有成员，专门去给类写一堆不必要的get接口，破坏类的封装性。

这里要提醒大家，如果一个二元运算符的第一个参数必须是别的类型，比如你要实现整数和我们自定义String类的相加做字符串拼接，比如写`123 + str`，这时候运算符的第一个操作数是int类型，它不可能是我们自定义类的成员，那这时候你就完全没办法把这个运算符重载成我们自定义类的成员函数，这时候就必须把这个运算符实现成全局的普通函数，这是很典型的场景。

然后大家可以记住一个比较方便的经验准则：所有的一元运算符，最好都实现为类的成员函数；像赋值`=`、下标`[]`、箭头`->`、函数调用`()`这几个运算符，是必须要作为类的成员函数来实现的，你不能把它们做成全局函数；剩下的其他的二元运算符，最好都把它们实现为全局的非成员函数，也就是普通函数，这样你的代码可扩展性会更好，尤其可以支持运算符第一个参数是其他类型自动转换的场景。

接下来我们讲参数传递的准则：如果你现在要传的这个E，它本身是个很小的内建类型，比如int、char这种，那你直接传值就没问题，拷贝的代价几乎可以忽略。但是如果你要传递的是一个类的对象，而且你传递它的目的，不是为了去修改这个传入的实参本身，那你就一定要把这个参数声明为const引用，这样既可以避免不必要的对象拷贝开销，也可以保护传入的实参不会被意外修改。如果你的目的就是要修改传入的实参，那你直接传普通引用就可以，不用加const，就像我们做赋值操作传参的时候的场景。

我们先稍微暂停一下，大家现在有什么问题可以随时提，刚才我们讲的这些部分有没有没听懂的地方？

好，我们接着往下讲。刚才说完了参数怎么传，接下来讲运算符重载的返回值要怎么选：你要根据你重载的这个运算符原本的语义，来选择合适的返回类型。比如你重载加法运算符，加法本身是不会修改两个操作数的，它会生成一个全新的临时对象作为运算的结果返回，所以你就返回一个新的类对象的值就可以。但是如果你重载的是`+=`这种复合赋值运算符，它本身是要修改左操作数本身的，而且我们本来就支持连续写`a += b += c`这种链式调用，所以你这时候就不能返回临时对象，必须返回当前对象的引用。而像`==`、`>`、`<`这种关系逻辑运算符，它们的返回值就应该是布尔类型，也就是bool，以前老标准用int也可以，但现在规范里都返回bool。

接下来我们来看不同运算符的标准重载原型：首先是加法这种二元算术运算符，标准的原型应该写成`const T operator+ (const T& a, const T& b)`，它返回的是const修饰的临时对象，防止你写出`a + b = c`这种没有意义的非法代码。然后复合赋值运算符`+=`的原型，就应该是`T& operator+= (T& a, const T& b)`，它直接返回左操作数的引用。然后下标运算符`[]`，它应该返回元素的引用`T& operator[] (int index)`，这样我们才能支持写`a[i] = 5`这种对返回值赋值的操作。然后取地址运算符`&`，我们一般不需要自己去重载它，编译器默认生成的版本就完全够用了。

接下来专门讲自增运算符`++`和自减运算符`--`的重载，这里有个很关键的问题：我们怎么区分前缀的`++`运算符，和后缀的`++`运算符？这两个运算符的名字是完全一样的，语义完全不同，一个是先自增再返回值，一个是先返回旧值再自增。C++的编译器给我们规定了一个非常巧妙的约定：我们在重载后缀的自增运算符的时候，给它多塞一个没用的int类型参数，这个参数永远不会被用到，写的时候甚至可以不写参数名。这样编译器看到你写的`operator++()`没有参数，就知道你这是前缀自增，看到你写的`operator++(int)`带一个没用的int参数，就知道这是后缀自增。

那我们具体来看实现：首先前缀自增的成员函数版本，我们直接把当前对象的内部值加1，然后返回当前对象的引用就可以，`this->value += 1`之后直接`return *this`。而后缀自增的实现就不一样了，因为我们要先返回自增之前的旧值，所以你要先基于当前对象拷贝一个临时的副本，然后再把当前对象自己的值加1，最后你把之前拷贝的那个存了旧值的临时对象返回就可以了。这也是为什么大家都说前缀自增比后缀自增效率高的原因，前缀自增根本不需要生成额外的临时对象，而后缀自增必须拷贝一个临时对象存旧值，开销更大。所以我们写循环的时候，能用前缀`++`就尽量用前缀`++`。

然后我们就可以来使用重载好的自增运算符了，比如说我们定义好了自己的自定义类`MyInt a(5);`，那写`++a`，调用的就是前缀自增的版本，它直接返回a的引用，a本身变成6。而写`a++`的时候，就调用带int参数的后缀版本，它返回的是之前值为5的那个临时对象，这时候a本身已经变成6了。大家可以看到，我们之前说的前缀自增效率比后缀高，本质就是少了一次临时对象的拷贝构造和析构的开销，尤其当你的自定义类很大的时候，这个性能差距会非常明显。

接下来我们讲关系运算符的重载。大家完全没必要把所有六个关系运算符`==`、`!=`、`<`、`>`、`<=`、`>=`全部手动写一遍，这样太浪费时间了。按照规范，你只需要先实现等于运算符`==`和小于运算符`<`这两个就够了，剩下的所有其他关系运算符，都可以直接基于这两个已经写好的运算符来实现，完全不用重复写逻辑。比如`!=`，就直接返回`!(*this == b)`，大于`>`的话就直接返回`b < *this`，小于等于`<=`就直接返回`! (b < *this)`，大于等于`>=`就直接返回`! (*this < b)`，这样你只用写两份核心逻辑，剩下的四个运算符全部复用之前的代码，不容易出错，维护起来也方便。

然后我们说下标运算符`operator[]`，它有个硬性要求：这个运算符必须是类的成员函数，它只能接收一个参数，就是你要访问的下标索引。如果你把我们的自定义类模拟成一个数组的话，你调用`obj[i]`的时候，它本质上就是调用`obj.operator[](i)`，所以这个运算符的返回值一定不能是普通的值，它必须是一个左值，也就是返回引用，这样你才能支持写`obj[i] = 新值`这样的操作。当然这里要提醒你，最好在`operator[]`里面做下标越界的检查，防止用户传进来非法的下标，导致程序访问非法内存直接崩溃，你可以抛出异常或者做报错处理，就像标准库里面的`std::vector`的`at`函数那样的安全检查。

接下来我们要区分清楚复制初始化和赋值操作这两个完全不同的操作。很多初学者很容易把它们搞混：写`MyType a = b;`这个操作根本不是调用赋值运算符，它是一个初始化操作，它用已有的对象b去构造一个新的对象a，调用的是拷贝构造函数。而如果我们已经提前定义好了`MyType a;`，然后之后再写`a = b;`这才是赋值操作，它调用的是赋值运算符重载，把b里面的值覆盖到已经存在的对象a里面。这两个操作是完全不一样的阶段，千万不要搞混。

这里还要注意一个非常重要的点：如果你自己没有在类里面显式地写赋值运算符重载的话，C++的编译器会自动偷偷地帮你生成一个默认的赋值运算符函数，这个默认生成的赋值运算符，做的事情就是把源对象里面所有的非静态数据成员，逐个拷贝赋值给目标对象的对应成员。很多人就觉得那我就不用写赋值运算符了，编译器自动给我生成的肯定没问题，但这个默认生成的版本在很多情况下是完全错误的，比如你的类里面有成员是指针，指向堆上分配的动态内存的时候，默认的逐成员赋值就只会拷贝指针本身的值，导致两个对象的指针指向同一块堆内存，最后析构的时候两次释放同一块内存，程序直接崩溃，也就是大家常说的浅拷贝问题。

所以赋值运算符重载必须作为类的成员函数来实现，如果你没有自己提供这个函数，编译器就会像自动生成默认构造函数、拷贝构造函数那样，偷偷给你生成一个默认版本。你自己实现赋值运算符的时候，一定要记得先做一个自赋值检测：判断你是不是把对象自己赋值给自己，也就是`if (this == &rhs) return *this;` 如果用户不小心写出`a = a`这种代码，你直接什么都不做就返回，避免你先释放了自己指向的动态内存，之后要从自己这里拷贝数据的时候就已经找不到数据了，出现严重的野指针错误。写完自赋值检测之后，你再去释放当前对象自己原来申请的动态资源，然后从右边的源对象那里把数据深度拷贝过来，最后别忘了`return *this`，支持赋值的链式调用。

我们可以写一个赋值运算符的骨架版本，来给大家演示一下基本结构：首先返回值是当前类的引用，然后参数是const引用传入右边的对象。首先第一步就判断this的地址是不是等于传入的rhs对象的地址，如果相等的话，直接返回`*this`。如果不是自赋值，我们先把当前对象之前申请的动态内存释放掉，然后再去rhs里面拿到它的长度，申请新的对应大小的内存，把rhs里面的内容逐个拷贝过来，最后返回`*this`就完成了。这里必须做的自地址判断非常非常重要，如果你忘了写，一旦出现自赋值的场景，你的程序直接就出现未定义行为，很容易直接崩溃。

好了，今天我们讲的所有这些内容，都是C++运算符重载最核心的规范和最佳实践，大家写代码的时候严格按照我们讲的这些准则来，就能避开90%以上运算符重载相关的常见错误。

我们继续来看之前的内容，首先讲赋值运算符的基础实现骨架：
每一个成员变量，我们都要把右侧赋值源对象里对应成员的值，赋值给当前目标对象的成员，同时第一步要先判断当前对象的地址和传入赋值参数的地址是否相等，避免自我赋值带来的错误，最后返回*this，这就是赋值运算符最基础的实现逻辑。

对于那些内部动态申请了内存的类，你必须自己显式自定义赋值运算符，否则编译器默认生成的浅拷贝赋值会直接导致内存泄漏、重复释放这类严重的问题。如果你完全不希望这个类支持赋值操作，你可以显式把赋值运算符声明为private权限，从语法层面直接禁止外界调用它。

接下来我们讲自定义流提取运算符，你需要把它定义为全局的友元函数：第一个参数是istream的引用，第二个参数是你要读入数据的自定义类对象的引用，函数返回值也必须是istream的引用，这样才能支持连续的>>链式调用。函数内部你从输入流里读取对应数据，赋值给目标对象的成员变量，最后把流对象返回即可。

和提取运算符对应的是流插入运算符，它的逻辑非常相似：第一个参数是ostream的引用，第二个参数是要输出的自定义类对象，这里第二个参数通常用const引用传递，避免不必要的对象拷贝。函数内部把对象的各个成员输出到输出流中，最后返回ostream的引用，同样就可以支持连续的<<链式调用。

接下来我们学习自定义操纵符的写法，你只需要定义一个普通函数，这个函数接收ostream的引用作为参数，在函数内部实现你想要的自定义格式逻辑，最后返回这个ostream的引用。之后你就可以直接把这个函数名放到<<运算符的右侧，就像使用系统提供的endl一样，你的自定义操纵符就可以直接生效。

接下来我们介绍值类：这类类的设计目标是表现得和C++的内置基本数据类型完全一致，它们可以直接作为参数传入函数，也可以作为返回值从函数返回，你可以给它重载所有符合逻辑的运算符，最后它就可以像int、double一样直接参与运算和类型转换。典型的例子有复数类Complex、日期类Date、还有C++标准库的String字符串类，这些都是非常经典的值类设计。

接下来就和值类相关，我们来讲解用户自定义类型转换：转换运算符是一种特殊的类成员函数，它可以让当前类的对象隐式转换为另一个指定的目标类型，C++编译器会在合适的时机自动调用这个转换函数，完成隐式转换。但这种自动隐式转换很多时候会触发完全超出预期的类型转换逻辑，带来大量很难调试的bug。

首先要注意，所有只接收单个参数的构造函数，本身就自带隐式转换的能力：比如你写了一个只传入int作为参数的Complex构造函数，编译器就会自动把一个int值隐式转换成Complex的临时对象，很多时候这完全不是我们预期的行为。
所以C++11之后新增了explicit关键字，你在单参数构造函数前面加上explicit修饰，就可以完全禁止这个构造函数被用来做隐式转换，所有的类型转换都必须由开发者显式写出，彻底避免编译器偷偷做隐式转换带来的各种诡异问题。

转换运算符的语法非常特殊：你在类里直接写operator 目标类型T()，它不需要声明返回值类型，也不需要传入任何参数，只需要在函数内部返回对应T类型的结果。之后编译器在需要把当前类对象转成T类型的时候，就会自动调用这个转换运算符，不需要你显式写任何转换代码。
所有自定义转换运算符的通用形式都是operator T()，T可以是任意合法的类型描述符，它没有返回类型声明，不需要参数，函数体内返回一个T类型的结果，编译器就自动帮你完成对应的隐式类型转换。

但要注意，C++的类型转换体系里，编译器最多只会自动帮你执行一层用户自定义的隐式转换，如果你需要先做第一层自定义转换，再做第二层自定义转换，编译器不会自动处理，这时候就会直接报编译错误，你必须手动显式完成两次转换。

最后我们来讨论一个很重要的问题：你到底应不应该在你写的类里使用这些隐式转换功能？绝大多数情况下答案都是“不推荐”，隐式转换很容易在你完全没预料到的场景下被触发，带来很难排查的逻辑bug和不必要的性能损耗，甚至导致完全意料之外的重载函数匹配结果。如果你确实需要从当前类得到某一个其他类型的返回值，更推荐你显式写一个普通的成员函数，比如类似toDouble()这样明确命名的函数，让使用者主动显式调用它来获得转换结果，这样代码的可读性更强，也彻底杜绝了意外隐式转换带来的所有隐患。
