# 第2章 指令级并行（ILP）

## 2.1 引言与体系结构设计的核心思想

[cite_start]指令级并行（Instruction-Level Parallelism, ILP）是现代微处理器设计的核心技术，其目标是让计算机在同一时间内执行多条指令，从而显著提升处理器的执行效率。在探讨具体的并行技术之前，我们需要回顾计算机系统设计的八大基本原则（Great Architecture Ideas），这些思想贯穿了体系结构发展的半个多世纪 [cite: 1066]。

1.  [cite_start]**面向摩尔定律设计**：芯片上的晶体管数量大约每18-24个月翻倍，架构师必须预见系统设计完成时的技术水平 [cite: 1071, 1072]。
2.  [cite_start]**利用抽象简化设计**：通过分层抽象，隐藏底层细节，为上层提供简洁的模型 [cite: 1074, 1076]。
3.  [cite_start]**加速经常性事件（Make the common case fast）**：这是阿姆达尔定律的体现，通过改进频繁发生的场景来获得最大的性能收益 [cite: 1077, 1079]。
4.  [cite_start]**通过并行提高性能**：在指令级、进程级等多个层面实施并行操作 [cite: 1080, 1082]。
5.  [cite_start]**通过流水线提高性能**：将任务分解为阶段，使多个任务能在不同阶段同时执行 [cite: 1086, 1087]。
6.  [cite_start]**通过预测提高性能**：在结果确定前通过猜测（如分支预测）提前执行指令 [cite: 1089, 1091]。
7.  [cite_start]**层次化存储结构**：利用存储的局部性原理，构建由快速、昂贵的小容量存储到慢速、廉价的大容量存储的层次结构 [cite: 1092, 1093]。
8.  [cite_start]**通过冗余提高可靠性**：引入冗余组件以检测并纠正故障 [cite: 1095, 1096]。

值得注意的是，现代体系结构教材（如第七版）在上述基础上更加强调了**安全性（Security）**。由于现代处理器大量使用乱序执行和缓存预测等机制，攻击者可能利用缓存访问的时间差进行侧信道攻击（Side-channel attacks），这使得安全性成为了系统完整性中不可或缺的一环。

## 2.2 流水线技术回顾

[cite_start]流水线技术是实现指令级并行最基本、最重要的方法之一。通过将指令的执行过程划分为多个步骤，可以让多条指令在时间上重叠执行 [cite: 1087]。

### 2.2.1 流水线的基本概念
[cite_start]流水线类似于现代化的工业洗涤过程：洗涤、烘干、折叠、收纳四个阶段可以并行工作 [cite: 1127]。在理想情况下，一个5级流水线每时钟周期可以进入一条新指令，并在末端产出一条完成的指令。为了支持这种重叠执行，硬件上需要在各流水段之间引入**流水线寄存器**。这些寄存器用于存放中间处理数据和控制信号，虽然增加了硬件设计的复杂度和开销，但通过提升系统的**吞吐率（Throughput）**显著增强了整体性能。

### 2.2.2 流水线冲突（Hazards）
[cite_start]流水线并不是总能完美运行。当某些情况阻止指令在预定的时钟周期内执行时，就产生了**冲突（Hazards）** [cite: 1215]：
* [cite_start]**结构冲突（Structural Hazards）**：硬件资源忙，无法支持指令的并行执行（如单端口存储器同时面临取指和访存需求） [cite: 1216, 1217]。
* [cite_start]**数据冲突（Data Hazards）**：当前指令依赖于前一条指令的执行结果，必须等待数据就绪 [cite: 1223, 1224]。
* [cite_start]**控制冲突（Control Hazards）**：指令执行流程受分支指令影响，在确定分支路径前无法确定取指地址 [cite: 1225, 1226]。

## 2.3 指令相关性：程序固有属性

在分析流水线冲突之前，必须理解**相关性（Dependence）**。相关性是程序的固有属性，它决定了指令之间潜在的执行顺序约束。

### 2.3.1 数据相关（Data Dependences）
[cite_start]当指令 $i$ 产生结果并被指令 $j$ 使用时，称指令 $j$ 与指令 $i$ 存在数据相关 [cite: 1161]。这种相关可能引发流水线中的**写后读（RAW）**冲突。例如：
```assembly
FLD      F0, 0(R1)
FADD.D   F4, F0, F2  ; FADD依赖于FLD的结果F0
```

### 2.3.2 名相关（Name Dependences）
[cite_start]名相关是指两条指令使用相同的寄存器名，但它们之间并没有实际的数据流动 [cite: 1167]。
* [cite_start]**反相关（Anti-dependence）**：指令 $j$ 写寄存器，而指令 $i$ 读同一个寄存器。这对应于**读后写（WAR）**冲突 [cite: 1172, 1236]。
* [cite_start]**输出相关（Output-dependence）**：指令 $i$ 和指令 $j$ 写入同一个寄存器。这对应于**写后写（WAW）**冲突 [cite: 1189, 1245]。

[cite_start]为了消除名相关带来的约束，通常采用**寄存器重命名**技术，将逻辑寄存器映射到不同的物理存储位置，从而允许指令乱序执行而不破坏逻辑正确性 [cite: 1173, 1190]。

### 2.3.3 控制相关（Control Dependences）
[cite_start]控制相关是由分支指令引起的。由于分支指令的存在，某些指令的执行取决于分支是否跳转 [cite: 1211, 1450]。

## 2.4 数据冲突的解决方案

为了减少流水线停顿，硬件和软件层面提供了多种手段。

1.  [cite_start]**重定向（Forwarding/Bypassing）**：通过在流水线内部添加旁路，将结果直接从运算单元引回到后续指令的输入端，无需等待数据写入寄存器堆 [cite: 1282, 1306]。
2.  [cite_start]**停顿（Stalls/Bubbles）**：当重定向无法完全消除冲突（如 Load-Use 场景）时，流水线必须插入“空泡”，暂停后续指令的执行 [cite: 1353, 1360]。
3.  [cite_start]**编译器指令调度**：编译器可以重新排列代码顺序，在 Load 指令及其使用结果的指令之间插入无关指令，从而填补流水线空隙 [cite: 1374, 1410]。

## 2.5 控制冲突与分支预测

[cite_start]控制冲突对性能的影响往往比数据冲突更为严重，尤其是在深度流水线和多发射处理器中 [cite: 1498, 1801]。

### 2.5.1 静态预测
[cite_start]静态预测在编译阶段完成，基于典型的分支行为。例如，预测循环末尾的后向分支（Backward branch）总是跳转，而预测 if 语句的前向分支（Forward branch）不跳转 [cite: 1517, 1518]。

### 2.5.2 动态分支预测
[cite_start]动态分支预测依靠硬件在运行时记录各分支指令的历史行为，以此预测未来的走向 [cite: 1530, 1531]。
* [cite_start]**1位预测器**：记录上一次是否跳转。缺点是在嵌套循环中会产生两次连续预测错误 [cite: 1813, 1821]。
* [cite_start]**2位预测器**：基于饱和计数器状态机。只有连续两次预测错误后才会改变预测策略，这能将预测准确率显著提升至 90% 以上 [cite: 1831, 1858]。

### 2.5.3 分支目标缓冲区（BTB）
[cite_start]仅仅预测分支是否跳转是不够的，还需要快速获得跳转的目标地址。**分支目标缓冲区（BTB）**是一种特殊的缓存，它存储了分支指令的 PC 以及对应的目标 PC [cite: 1917, 1997, 1998][cite_start]。如果在取指阶段 BTB 命中，处理器甚至可以在指令解码前就开始从目标地址取指，从而实现无延迟的跳转 [cite: 2000, 2011]。

## 2.6 设计哲学的演进：从 MIPS 到 RISC-V

[cite_start]在早期的 RISC 架构（如 MIPS）中，设计者引入了**分支延迟槽（Branch Delay Slot）**，要求分支指令后的指令无论分支是否跳转都必须执行，以利用跳转判定的等待时间 [cite: 1789, 1793]。然而，随着流水线变深和多发射技术的出现，单一延迟槽已难以填补空隙且增加了软件负担。因此，RISC-V 架构果断舍弃了延迟槽，转而完全依赖高效的动态硬件分支预测机制，这种“极简硬件”与“动态调度”结合的思路已成为现代高性能处理器的主流。

### 综合练习：流水线冲突分析
[cite_start]考虑以下代码段 [cite: 2101, 2102]：
1. `FADD.D  R1, R2, R4`
2. `FADD.D  R2, R1, 1`
3. `FSUB.D  R1, R4, R5`

**分析：**
* 指令1与指令2之间存在 **RAW** 冲突（数据相关 R1）。
* 指令1与指令2之间存在 **WAR** 冲突（反相关 R2）。
* 指令1与指令3之间存在 **WAW** 冲突（输出相关 R1）。
* 指令2与指令3之间存在 **WAR** 冲突（反相关 R1）。

[cite_start]在简单的5级顺序流水线中，RAW 冲突可以通过重定向解决，而 WAR 和 WAW 冲突通常不会导致停顿；但在乱序执行流水线中，必须通过寄存器重命名等手段来处理上述名相关，以确保程序逻辑的最终正确性 [cite: 2103, 2104]。
