## 三、局部性原理

然后呢关于这个 Memory Hierarchy 这部分呢，其实还有一个我们一直用到的原理，就是时间局部性和空间局部性。那这个呢是在每一个层级都比较适用的。这跟我们程序存储的方式、运行的方式是有关的，对吧。我们不希望说我们从不同的地址去，这儿要拿一点那儿要拿一点，这个访问起来，我们包括寻址包括传输都是需要时间的。那为了提升 Performance，也就是说我们所有的设计最终都是为了提升 Performance 对吧，我们不希望说增加了一个设计，然后使得我的程序的运行又变慢了，这个是不愿意看到的。

那怎么让它快呢？最好是我这一行程序读完以后，接下来我就在物理空间上紧挨着它读下一行，这个地址空间是连续的，对吧，在物理上是连续的。然后我实际运行起来是逻辑程序，实际上是抽象层，那这个抽象层是从连续的物理空间来读，这个当然就比较快。所以这个是空间上的局部性。那程序也满足这样的局部性，就是我从某一个物理空间去读一条指令，那我接下来的 next 指令很可能就是紧挨着的下一条，这个就是空间上的局部性。

那时间上的局部性呢，就是某一些数据，我刚刚访问过，很可能我在接下来的一个很短的时间内，我又要访问这个数据。就某些数据某些变量会被我们反复地使用。对啊，比如说在循环程序当中，那某一些语句我们会反复地多次地使用，所以这是时间上的局部性。

那程序的运行呢是满足这两个局部性原理的，就是时间局部性和空间局部性。那这也是经过很多程序的验证而得到的结论。那接下来我们会讲到的和 Memory 和 Cache 有关的内容都会遵循这两个局部性的原理。我们也是利用了这个局部性的原理来做了很多优化的设计。

## 四、Cache 的基本概念

那 Memory 是什么呢？我们在前面做流水线设计的时候，实际上已经大量地提过，访存指令比如说 Load Store 这样的访存指令，它需要去访问内存。那实际上仅仅有内存呢还是不够的，对吧。那我们在这个处理器他去访问内存的过程当中呢，实际上我们离这个处理器越近的，那可能是一个更小的、局部的这样的一个 Memory，然后逐渐离它越远，我们需要有更大的 Memory 来做备份，比如说我们有永久存放的数据做备份。

所以他整个的层次呢就是离这个 CPU 越近，那么这个容量是越小，但是他的每位的价格就越高。然后离 CPU 越远呢，他每位价格也越低，但是整体的容量就越大。那所以呢在离 CPU 很近的地方呢，我们通常用 SRAM，也就是我们用的这个静态随机访问。那这种介质呢，我们一般用来做 Cache。那 DRAM 呢我们是用来做 Memory，就是离他更远一级。然后呢以及到这个 Disk，所以这是整体的他这个层次设计的这个逻辑，主要考虑到的呢是数据的传输和数据的存储。

好，那么我们知道在 CPU 内部呢有 Register，那前面我们大量的这些指令，实际上我们对这个 Register 也比较熟悉了。然后呢我们也知道这个 Load Store 指令要访存，对吧。所以我们对 Memory 也是了解的。所以我们前面知道的介质有什么呢？有 Register、Memory，那当然还有我们的磁盘，Storage。所以呢他们中间呢通过这些总线连接，然后可以传输数据。比如说我 CPU 要的数据，我就从上一级一点一点传过来。

那当我们现在所有的数据都在 Storage 上面，那 CPU 要访问它的时候怎么办呢？那么就是从 Storage 就会掉到 Memory。因为我 CPU 所有访问的数据都在 Memory 上面或者是在 Register 上面。那我反正不能直接从 Storage 上面去访问，他没有这样一个直接访问的通路，对吧，所以都需要通过 Memory。

好，那么随着这个技术的发展，当 CPU 的速度很快，然后 Disk 速度肯定是慢的，Disk 非常慢。然后呢这个 Memory 的速度呢也不够，也跟这个 CPU 的速度之间的差距越来越大。就出现了这样的一个 Performance Gap。那这个 Performance Gap 呢，我们就需要有一个 Buffer 来填，对吧。这个就很像我们设计流水的时候，当我们这个取指后面跟执行部件之间存在很大的速度的差异，我们怎么办呢？我们放一些取指的 Buffer 对吧，那么让这个执行部件不是直接从 Memory 去拿指令，而是从这个 Buffer 当中去拿指令，对吧。然后写的时候也是一样，我们有写出的 Buffer。那他执行完之后写到这个 Buffer 里面，那他就可以一直写，写得很快。那然后再从这个 Buffer 里面呢，再传到 Memory，就中间加了这样的一个 Buffer 对吧。

所以我们前面会讲到，在 CPU 设计的时候用这些 Buffer。所以这里呢也很自然地想到在 CPU 和 Memory 中间，我们加一级这个 Buffer，这个 Buffer 呢我们就叫做 Cache。那么 Cache 呢它就处在 CPU 和 Memory 中间的这个位置。当然针对不同的场景呢，我们对他的大小要求是不一样的。比如说我这个 Cache 到底设计多大呢？那不同的计算机会对它有不同的要求，会根据你这个计算机使用的场景，他会有不同的要求。

比如说对于我们这个个人使用的 Mobile 这样的设备来讲，对吧，那一般呢我们设两级的 Cache，就是 Cache 一级还不够。我们有 L1、Level 1，Level 2 两级 Cache。那么 L1 呢就离 CPU 更近，它一般就会更小，64K。那么可能 L2 呢就是 256K。然后他们的这个访问的速度也是离 CPU 越近速度越快，离 CPU 越远呢就速度越慢。所以这个我们从这个红色的箭头可以看到，从左到右，他的容量是逐渐增大的，当然他的速度呢是逐渐变慢的。所以访问的这个时间是逐渐变多的，所以是从左到右的这些层级。这个当然对于一个 Mobile 来讲，那么他从这个量级来看，就是百位量级的 GB 就够了。所以这是看对 Mobile 的要求。

那当然对于一个我们的 Desktop，就是我们讲的桌面计算，就是我们的笔记本、台式机这样的，那我们就用到了 L3 Cache，就是把 Cache 分为了三个层级，L1、L2、L3 三级。我们看到这个 L3 Cache 就基本上就是兆 B 了，就是几兆 B。然后当然这个现在也可以很多个 GB，也就是在 GB 这个层级就够了。所以从左到右呢，也是这个容量不断地增大，然后速度不断地减慢，这是从左到右的发展情况。

当然比如说对于 Server，那么他就需要更大的容量，他需要更大的容量，那么需要更高的稳定性。所以我们看 Server 的这个 Memory 是比较大的，像我们个人级这个 Memory 也是在这个几十个、十几个 GB 就可以了。但是到 Server 的这个 Memory 就要到几百个 GB。所以他是从左到右也是不断地增大。那么它的 Disk 也是更大的，因为它所适用的场景是不一样的。

所以我们看针对不同的使用的场景，那么我们可能对具体的设计来讲，我们可以采用不同的量级。但是它的架构呢依然是我们看到的这个层级架构，就是 Memory Hierarchy 整体的层级架构都是一致的，都是采用的同一套的模型。

那么到底这个中间的这个 Buffer Cache，他到底是什么样的，对吧？我们看到不同类型的计算机系统其实都需要 Cache，那么可能需要这个 L1、L2 级 Cache，或者三级 Cache 或者更多的 Cache。那么 Cache 到底是什么？Cache 到底是什么，以及他到底采用什么样的方式来组织。那我们知道呢这个我们可以把一些程序、一些数据放在 Cache 上面。那么他会让我们看到 Cache 上面有一些数据的换入、换出，对吧。那到底什么样的数据要换入到这个 Cache 当中去。那以及呢我在 Cache 当中怎么找到我想要访问的数据。对，我访问这个数据到底在哪儿，对吧，那这个数据我要访问的我是不是能够精准地找到。那既然我要找我是不是能够很快速地找到，我不能找得很慢，我找得很慢，我加这个 Buffer 我就得不偿失了啊。那能不能去做一些优化，对这个 Cache 当中我所采用的这种机制，我最终是为了加速，我为了平滑 CPU 和 Memory 之间速度的差异，我加了这个 Cache。那这个 Cache 怎么样组织，他上面所存放的数据采用什么样的形式，那 Cache 用什么样的结构来存这些数据。那所有这些问题呢，我们就带着这些问题来学习 Cache 有什么样的策略。我们可以使用 Cache 的时候，让他速度更快，表现更好，让他能够更利于我们去加速程序的运行。

好，那这些问题都是在我们要学习完 Cache 之后大家能够解答出来的问题。

那什么是 Cache 呢？他当然有一个定义，那么对 Cache 的定义呢是认为他是一个很隐蔽的、但是可以存放一些 something 的一些地方，对吧，就是可以一个隐蔽的存放数据的地方。那实际上正因为它的隐蔽性，也会带来他的安全隐患，对吧。那么我们怎么样在这个隐蔽的地方存放呢？也就是他并不是显性的存在，那么他一直在，但是我们在前面讲这个流水线的时候，我们都没有提过他，对吧。我们就觉得我需要保存，那我就 Load Store 指令从 Memory 拿数据就好了，我们就根本没有提 Cache。但 Cache 他其实一直都在的，一直都是在的，只是他经常容易被忽略掉。但是我们实际上会用到它，那为什么会用到它呢？实际上也跟 Memory 的发展有关。他不仅跟 CPU 和 Memory 速度的发展有关，还跟什么有关呢？大家想还跟 Memory 的这个大小的发展有关，对吧。随着我们刚才看到的这个器件硬件技术的发展，然后 Memory 也越来越大了。那一开始 Memory 比较小的时候，那可能就跟现在 Cache 这么大的时候。那我们直接从 Memory 当中去拿，可能按照一定的算法，我就去拿这个数据了。但是 Memory 他也越来越大，那我在里面找也不太好找，那我就需要放一个更小的存储的空间，然后从 Cache 当中去找。

那我怎么样类比这个例子呢？就比如说你可能一开始只有一个书柜的书，那么这个书柜就是你的 Memory，你有一个书柜的书。那你从这一个书柜里去拿这些书，你大概扫一扫，怎么扫呢？你其实也是按层扫，你大概知道你这个书可能在第几层。那你在这个第几层这个书柜里面就把这个书拿出来了，这个是很好找的。那突然随着你不断长大，你又增加了很多书柜，你现在可能有一屋子的书，那你这个内存就不断地变大。假如说这个书柜就是你的 Memory 的话，你就不断地变大，那你找就很难找。那你总有一些要经常看的书吧。好，那你就把经常看的书呢，就放在你的书桌上。这个书桌的作用是什么呢？就是我坐在这儿的时候，我随时可以拿到，我一伸手就拿到了。我比跑到这个书柜这去拿要更快吧。那么这个书桌就是你的 Cache。所以你这个书桌上能放的书的容量是有限的，它不像书柜，我可以不断地买书柜。那我这个书桌就这么大，当然我这个书桌也可以变得更大，但不管多大，它这个容量也就是有限的。好，那我在这个书桌上就可以放一些我常看的书，我常看的书我随时看，我就可以拿了。

那我怎么知道我常看哪些书呢？就比如说计算机，我怎么知道我要访问哪一些资源呢？这里就是局部性原理了。局部性原理就是我昨天看的这本书，我觉得我今天可能还要看，它是不是有这个连续性啊，时间上的连续性，有时间的局部性。那我刚看的这本书，可能我刚看了上册，我就推测可能要看下册，所以他是空间上的局部性，它都具备。所以我在桌上放的这些书，很可能我大概率都要经常地去看。但是也有一些情况就是我今天要看的书，我发现我这个书桌已经满了。但是呢我现在要看的书还不在我的书桌上，这个时候怎么办呢？我就要去书柜里拿了，对不对？你要去 Memory 当中拿。那拿这个时候假设说我这个书桌容量只能放十本书。我们假设因为 Cache 容量是有限的，假设我只能放十本书，但是我这十本书已经满了，还没有我要看的书，我是不是要去书柜里拿。拿完之后，这个书是不是要放在这儿，但是是不是要从原来的十本书当中要替换走一本吧。那替换走哪一本呢？这就是 Cache 的替换策略，对吧。就是你要替换走 Cache 当中的哪一个数据。你要拿走哪一本书，这十本书你选你拿走哪一本？如果是你选，你拿走哪一本，你觉得。这就是 Cache 当中我们要选择的策略。大家带着这个问题，我们后面都会讲到，我们到底要替换哪一本，这个我们到后面讲到这个算法的时候再去说。
