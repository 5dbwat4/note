# 分页机制（Paging）课程逐字稿

我们今天继续讲解分页机制。上周我们讲解了分页的基本概念，这周我们将继续深入，争取在本周或下周的第一次课上把分页这一部分的PPT全部讲完。我们上次讲到，分页的核心思想是将物理地址空间划分为固定大小的块，称为帧（Frame），同时将逻辑地址空间划分为同样大小的块，称为页（Page）。通过这种划分，我们可以将进程的逻辑地址放到碎片化的物理内存中，从而实现非连续的内存分配。

为了实现这种地址转换，我们需要一个关键的数据结构来存储从逻辑地址到物理地址的映射关系，这个数据结构叫做页表（Page Table）。上次我们也提到了页表中的内容，页表项中存储的是物理帧号（Frame Number），而不是逻辑地址或页号。那么帧号是如何计算的呢？我们将在后续的讲解中详细说明。

采用分页机制可以极大地缓解内存碎片化的问题。具体来说，分页没有外部碎片（External Fragmentation），因为所有的页和帧的大小都是固定的。但是，分页仍然存在内部碎片（Internal Fragmentation），不过这种内部碎片只出现在最后一个页或最后一个帧中，因此其比例被极大地缩小了。随着内存价格下降和容量增大，现在大家更倾向于使用较大的页或帧。

接下来，我们围绕页表展开讲解。页表存储的就是从逻辑页到物理帧的映射关系。这是一个典型的页表示例，大家需要掌握页表的填写方法。比如，页0被映射到帧14，所以在页表项0的位置存放的是帧号14；页1被映射到帧13，依此类推。

现在我们来回顾逻辑地址的定义。我们已经多次定义逻辑地址，这次是第三次。逻辑地址由两部分组成：第一部分是页号（Page Number），第二部分是页内偏移量（Offset）。通过页号，我们可以索引页表，找到对应的物理帧号，然后加上偏移量，就能确定具体的物理地址。

在32位架构中，如果页大小为4KB，我们来计算一下页号和偏移量分别占多少位。因为最小的寻址单位是字节，4KB等于2的12次方字节，所以偏移量占12位。总共32位的地址空间，因此页号占20位。也就是说，在32位架构下，逻辑地址的高20位是页号，低12位是偏移量。

MMU（内存管理单元）在这个过程中负责地址转换。它接收逻辑地址，按照20位和12位的划分，用页号作为索引去页表中查找对应的物理帧号，然后将帧号和偏移量组合成物理地址。那么，如何从逻辑地址中提取页号呢？很简单，将逻辑地址右移12位即可。如何从物理地址中提取帧号呢？同样，将物理地址右移12位。当然，这些计算的前提是物理地址和逻辑地址的长度相同，我们在当前阶段可以这样假设。

我们来看一个具体的例子。假设有页0、1、2、3，它们分别映射到不同的物理帧。现在给出页表中的映射关系，让大家自己画出页表的结构。页表中存储的是帧号，而不是逻辑地址或页号。此外，页表项中除了帧号，还包含一些标志位，用于标记映射的有效性以及访问权限等。

关于页表的实现方式，早期的计算机科学家面临一个问题：如何用软件或硬件来实现页表？最直接的方式是将页表存储在一组寄存器中。这种方式的好处是访问速度快，但缺点是寄存器数量有限，限制了页表的大小。此外，在上下文切换时，需要保存和恢复这些寄存器的内容，开销较大。

另一种更优的思路是，既然称为页表，我们就可以把它当作一个数组存放在内存中。这时，我们只需要使用一个页表基址寄存器（Page-Table Base Register，PTBR）来指向页表在内存中的起始位置。在RISC-V架构中，这个寄存器叫做SATP（Supervisor Address Translation and Protection）；在ARM架构中，它叫做TTBR（Translation Table Base Register）；在x86架构中，它叫做CR3（Control Register 3）。可以看到，现代主流芯片都提供了对分页机制的支持。特别是RISC-V和ARM，由于出现时间较晚，不再支持分段机制，因此只支持分页。

将页表放在内存中后，每次数据访问都需要两次内存访问：第一次访问页表获取帧号，第二次访问物理地址获取实际数据。与之前只需一次内存访问相比，效率下降了50%。为了优化这个问题，我们引入了TLB（Translation Look-Aside Buffer），这是一种专门用于缓存地址转换结果的缓存。TLB中缓存了最近使用的页号到帧号的映射关系。当需要地址转换时，首先查找TLB，如果命中，则无需访问页表，只需一次内存访问；如果未命中，则仍需两次内存访问。TLB通常采用相联存储器（Associative Memory）实现，支持并行查找，因此速度很快，但容量较小。

TLB的命中率对系统性能影响很大。例如，假设内存访问时间为100纳秒，TLB命中率为80%，则有效访问时间（EAT）为0.8×100 + 0.2×200 = 120纳秒，性能下降了20%。而实际系统中，TLB命中率通常高达99.99%，有效访问时间仅比单次内存访问慢不到1%，这充分体现了TLB的重要作用。

页表项（Page Table Entry，PTE）中除了帧号，还需要一些标志位。例如，有效位（Valid Bit）用于标记该映射是否有效；权限位用于标记页的读写执行权限。在32位架构中，页表项本身也是32位。前面我们计算出帧号需要20位，那么剩余的12位就可以用来存放这些标志位。在ARM架构中，为了方便，将帧号放在高20位，低12位存放权限位。RISC-V架构则是在中间选取20位存放帧号，剩余位存放其他信息，这也导致了实验中需要频繁进行移位操作。

分页机制还有一个优点，就是方便内存共享。例如，当多个进程运行同一个程序时，代码部分是只读的，可以在多个进程间共享同一份物理内存副本，而数据部分则是每个进程私有的。通过页表，我们可以将相同的物理帧映射到不同进程的页表中，从而实现共享。这种共享在分段机制中是无法实现的，因为分段机制下进程直接访问物理内存，缺乏灵活的映射层。

我们还可以观察实际系统中共享库的情况。例如，C标准库（libc）在系统中只占有一份物理内存，但被映射到几乎所有进程的地址空间中。这是因为libc的大小有数兆字节，如果每个进程都单独复制一份，内存将很快耗尽。

接下来，我们讨论页表的结构。在32位架构、4KB页大小的前提下，单级页表需要多少内存呢？总共有2的32次方除以2的12次方，等于2的20次方，即约100万个页表项。每个页表项4字节，因此页表大小为4MB。而且，这个页表必须是物理连续的，因为页表本身也需要寻址。此外，每个进程都需要自己的页表，这意味着即使一个进程什么也没做，仅页表就要占用4MB内存。如果系统总内存为4GB，最多只能支持约1000个进程，实际运行几十个进程就可能耗尽内存。

如何优化单级页表的问题呢？一个关键的观察是，进程的地址空间通常存在大量空洞。例如，程序可能只访问了第一个页和最后一个页，中间的页表项都被浪费了。为了解决这个问题，早期的计算机科学家想到，既然页表可以放在内存中，那么页表本身也可以被分页。因为一个页面可以容纳1024个页表项（4KB除以4字节），我们可以将页表分成多个页面，然后使用另一个顶层页表来索引这些页面，这就是多级页表（Hierarchical Page Table）的思想。

以两级页表为例，第一级页表（顶级页表）的每个条目指向一个第二级页表的物理帧，第二级页表的每个条目则指向实际的物理帧。这样，顶级页表只需要4KB，第二级页表则根据需要分配。在刚才的最坏情况下（只访问第一个和最后一个页），我们只需要分配顶级页表、第一个第二级页表和最后一个第二级页表，总共仅需12KB，相比单级页表的4MB，大大节省了内存。

在多级页表下，逻辑地址的划分也发生了变化。在32位架构、4KB页大小的前提下，两级页表的逻辑地址被分为三段：页目录号（P1）占10位，页表号（P2）占10位，页内偏移占12位。其中，页目录号用于索引顶级页表，页表号用于索引第二级页表，偏移量则用于在物理页内寻址。

如果页大小改变，例如从4KB变为64KB，则偏移量变为16位（因为64KB等于2的16次方字节）。在32位架构下，剩余的16位分配给P1和P2。由于一个64KB的页面可以容纳16K个页表项（64KB除以4字节），即2的14次方，因此P1和P2需要能够索引最多16K个条目，所以P1和P2各占7位？实际上，在两级页表的设计中，通常采用均匀划分，这里我们需要重新计算。以64KB页大小为例，每个二级页表包含16K个条目，所以P2需要14位，而P1则剩下2位（32位减去16位偏移再减去14位P2）。但具体的位数分配取决于系统设计，此处不再展开。

在两级页表下进行地址转换时，首先从逻辑地址中提取P1，作为顶级页表的索引，找到对应的二级页表的物理地址；然后提取P2，作为二级页表的索引，找到对应的物理帧号；最后加上偏移量，得到最终的物理地址。这个逐级查找的过程称为页表遍历（Page Table Walk）。

在Linux系统中，以x86架构为例，CR3寄存器指向顶级页表的基址，地址转换过程遵循上述步骤。这种经典的地址转换方式要求我们熟练掌握。

32位架构的物理地址空间上限为4GB，而现代手机内存已普遍超过4GB，这就导致了32位系统无法充分利用大内存的问题。为此，引入了物理地址扩展（Physical Address Extension，PAE）技术，但随后整个架构逐步过渡到64位。在64位系统中，虚拟地址空间为2的64次方，但实际硬件通常只实现39位或48位寻址，例如ARM64支持39位或48位，分别对应三级页表（9+9+9+12）或四级页表（9+9+9+9+12）。
