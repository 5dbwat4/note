# 逐字稿：页表结构与内存管理

## 一、两级页表的引入与空间计算

前面我们讲到，这种两级页表的结构，我们把它放到一个页面里面。所以现在我们就有了两层页表结构，每一个配置都有一个entry指向它，我们把这些entry放到一起，就形成了一个两层页表结构。这就是我们怎么样来演化出来的。

那这样来说的话，这里面有几个问题。第一个就是顶层这个列表，它占多大空间？同学回答：一个页就是4K。那中间这一层的页表它占了多大空间？我们需要分情况来看。有同学回答“看情况”，没错，如果你的invalid bit是0，表示它是invalid的，它没有映射任何二级列表，那你就可以说我只用比如我访问了哪一个，我就分配了哪一个，它是按需分配的。为什么能省内存呢？就是因为按需分配，所以它就能省内存。

那最坏情况（worst case）下是多大呢？有同学回答4兆，正确。就524K，它分配满了就是4兆。接下来我们跟大家讲这个图，关于每一段有多少个bit——怎么来的呢？取决于它要索引多少东西，我们就给它分配多少个bit。

那offset是怎么来的？因为页面大小是4KB，offset必须是12位index，它才能对每一个字节进行索引，所以是12位，就是2的12次方等于4K。中间那一级（PT）为什么是10位？因为它要索引1K个entry，所以需要10位。前面那一级为什么也是10位？因为它也要索引1K个entry。

## 二、页表遍历（Page Table Walk）

有了这个之后，我们就要学会去walk page table了。当时给大家举了一个例子，这是x86架构，讲的是地址怎么walk的。拿到一个logical address，把前10位拿过来，用它算一个数，这个数就是在这个页表里的index，从0开始数，数到这个位置就知道这个entry了。这一个entry里边存的是frame number，指向下一个page table的base。这个base，我从中间这10个bit拿过来算成一个数，就到第二级页表的entry拿到它。拿到这个之后，里面这个就是frame number，拿过来跟我们offset一加就成了物理地址。

我们上次课让大家算了一个题，结果不尽人意，我们一起来算一下。这个题目大家是怎么来算的呢？拿到一个这样的地址怎么算？有同学回答的是对的。我们把PTBR（Page Table Base Register）这个寄存器里面的值拿出来，这里面其实是一个障眼法，这个数据没用，我本来还想通过这个base给你们搞一个更复杂的，后来想想算了，再搞复杂估计就挂了啊。

我们就直接看这个virtual address，它是十六进制的0xf2015202。先把这个东西写成二进制：F是1111，2是0010，0是0000，1是0001，5是0101，2是0010，0是0000，2是0010。只有32个bit。我们怎么样来算？就按10、10、12这样来分。前10位到这儿是一个数，中间10位到这儿是一个数，最后12位是一个数。

现在来算一下这个数是多少：一级页表的偏移是986，二级页表的偏移是21。这个下面这个呢也一样，你就把这个地址展开成32位，然后按照10、10、12这样来算。

## 三、64位架构的地址空间

32位讲完了，我们来看64位。上次跟大家讲到计算机硬件的发展，当时大家疯狂堆内存，在32位架构、64位架构还没有ready之前，大家就已经推出了8G内存了，这就出问题了。后面64位赶紧需求补上就上了。

64位里面，如果64个bit全部用来index，内存应该是多大？2的64次方。2的60次方叫Exa（艾字节），它能index 16EB这么大的内存，这个内存量太大了，一般都用不完。所以我们其实用不着把64个bit全用上，所以就用了39位和48位，现在又出了57位，AMD用的是48位。

39位能index多大内存？512GB。48位呢？256TB。57位呢？128PB。现在57位已经几乎不够用了，因为搞AI的内存特别特别大，256TB都不够了，所以只能演化出57位。

大家有没有好奇这个39是从哪来的？为什么是39不是38不是40？我们来算一下：假设还是4K的页面大小，最后的offset有多少位？4K页面大小，offset就是12位。前面我们有几级页表？假设这一级叫P3，P2，P1。PTE entry在64位架构下是64个bit，就是8字节。页面大小4K除以8字节等于512，512是2的9次方，所以这一级是9位。同理每一级都是9位。三个9加12等于39。那如果不够用了，再加一级又加个9，等于48。再不够用再加一级，就是57。所以39、48、57就是这么来的。

如果是39位，就是三级页表；48位就是四级页表；57位就是五级页表。

## 四、Linux中的页表命名

每一级的page table我们给它起个名。第一级在Linux里边叫PGD（Page Global Directory），全局目录。中间这一级叫PMD（Page Middle Directory），这一级叫PTE（Page Table Entry）。以后你说哪一级，我们一般不太喜欢说什么一级页表二级页表，因为你到底这个“一级”是指他还是他，很难说。所以我们一般说话比较准确，就用PGD、PMD和PTE。

那如果是四级页表，他俩之间又差了一层，叫PUD（Page Upper Directory）。五级页表呢，它也没词了，叫P4D。现在最多就是五级页表。如果三级页表的时候，Linux就把这两层给它叠起来了，具体怎么叠的可以去看source code。

## 五、三种页表结构

### 1. 层次化页表（Hierarchical Page Table）

我们终于讲完了第一种页表结构，也是现在最常用的。后面两种都是脑洞大开的，不常用，因为效率会有问题。

### 2. 哈希页表（Hashed Page Table）

哈希页表的思想有点像你们刚学过数据结构——页表存的不就是一个logical address到physical address的映射嘛，那把logical page number哈希一下，找到那个地方，把物理帧号放进去不就行了？哈希的位置就存这个frame number。哈希算法最大的问题是什么？有可能两个哈希结果一样，就冲突了（collision）。对于哈希一样的，就用一个链表，用pointer指向下一个。

哈希页表的优点：没有五级页表，就一个哈希方式，多干净。缺点：冲突了就有链表，链表一长效率就变低了。但最重要的不是这个，最重要的是这东西很慢——跟层次化页表只用访存相比，哈希页表得先算半天。虽然现在有硬件辅助，有硬件指令一条就可以把哈希算出来，但它依然比较慢，一般不是一个fast path。像我们用的那个加密算法，一条指令要7个cycle才能算完，哈希方式很慢，会成为瓶颈。

### 3. 倒置页表（Inverted Page Table）

第三个结构又有大聪明想了：我能不能把页表反过来？讲这个之前，先讲一个概念——虚拟地址空间和物理地址空间。64位上我们一般用39位的logical address，那logical address space是2的39次方，等于512GB。而物理内存一般多大？手机撑死16GB，远小于逻辑地址空间。

大聪明就想：我不从logical address来做映射了，我从physical address来做映射。因为物理地址的space远小于逻辑地址空间，除以页面大小4KB，要index的entry数量明显少很多。所以能不能做一个inverted page table——不从logical address指向physical，而是从physical开始指向logical？

每个entry里面存的是process ID和这个process里面那个frame对应的page number。听起来像是一个很brilliant的idea，但是每次要查映射时，我有process ID和page number，还有一个offset，我得把所有表过一遍做匹配，找到那个entry里面正好是PID加page number，才知道对应的frame number。

它的优缺点：效率有问题，因为每次要找就得把整个表从头到尾过一遍，最坏情况要读N次内存，N是physical frame的数量。所以它也没有被用上。

这里有个问题：为什么需要PID？PID是用来区分不同进程的。页表和进程的对应关系是什么？是一个进程对应一个页表，一对一，不能一对多。因为页表基址寄存器只有一个，你要有两个页表选哪个用？所以最干净的就是一对一。

那如果一个操作系统里有5个进程，还跑了一个内核，总共有多少套页表？6个页表——每个进程有自己的页表，操作系统内核也可以看成是一个进程，也得有自己的页表。你们实验三里还没有进程，只有一个操作系统，就只有一个页表。到lab4有user space概念了，user space进程的页表就可以跟内核的页表独立开了。

如果用的是inverted page table，它只有一个表，把所有物理帧都放进去，所有进程的都在这儿，统一去walk，就会很慢。

这三种页表结构我们就讲完了。大家有没有想过一个问题：如果TLB发生了一次miss，现在要去walk这个页表，是硬件去walk还是软件去walk？答案是硬件去walk。为啥不让软件？软件walk一级一级走太慢了，虽然硬件也是一级一级走，但比软件快得多。x86、ARM和RISC-V都是硬件做，MIPS是软件做，效率堪忧，所以现在没人用了。

## 六、交换分区（Swapping）

讲一个概念叫swap（交换分区）。当你内存不够了，就拿一块硬盘说这块硬盘就是内存的swap区，可以把一些不用的东西放到硬盘上——内存不够硬盘来凑。一旦发生了swap，电脑几乎就卡死了，动作特别慢。因为硬盘比内存慢得多，把硬盘当内存用，注定会很慢。很多时候你装了个Linux，卡了，其实它没挂还在跑，只是跑得特别慢，你以为它卡了。

如果你发现电脑卡了或者打开页面卡了，一般怎么办？A继续等，B杀了重新开，C重启——重启治百病嘛。大家看英剧《IT狂人》里面所有IT解决问题的方法就是问别人有没有重启电脑。

swap在mobile上被disable掉了。因为手机或固态硬盘有写寿命，最多只能写多少次，把硬盘当内存用会频繁写，就把硬盘搞坏了，得不偿失。所以在mobile system或SSD上swap就被disable了。

swap是一个比较老的概念。内存不够了，把一个进程swap out放到硬盘上，再把新进程加载进来，这叫swapping。加上swap因素后，context switch时间会大幅提升。假设swap time是2000ms，进程100MB，硬盘传输速度50MB/sec，swap out两秒，swap in两秒，光读写硬盘就4秒钟——context switch一下4秒钟，而你手机context switch一秒发生几百次，所以你以为它卡了，实际上没卡死，还在跑。

怎么样优化呢？不用把整个进程换出去，只换出一些页，换进来一些页就行。新进程只要把第一个页换进来让它先执行。但总的来说swap性能很差，现在在某些系统上直接就不用了。

## 七、Intel 32位与64位架构

我们来看在真实的CPU上，页表和内存管理是怎么做的。首先看Intel，计算机硬件发展史前面几十年就是Intel的发展史。32位架构叫IA-32（Intel Architecture 32），64位叫IA-64。

IA-32支不支持paging？支持。支不支持segmentation？也支持。IA-64已经不支持segmentation了。Intel有个比较好的地方是向前兼容（compatibility），老的Intel程序能在新架构上跑，所以虽然演化这么多年，最早那代有segmentation，后面就一直支持segmentation。

segmentation怎么支持的？每个segment 4GB，有多少个segment？16K个，其中8K是进程自己的（LDT），8K是全局共享的（GDT）。Intel的segmentation跟课本上讲的一模一样——selector其实就是segment number，到这里面找到base，加上offset就成了地址。我一直比较喜欢调侃Intel，这个公司的人比较死脑筋，课本上怎么说的，他就严格按照课本去造硬件，到现在造的一些安全机制跟别人发的论文几乎一模一样，一点优化都没有。

segmentation做完之后产生了一个地址，这个地址不能再叫logical address了，叫linear address（线性地址）。下一步还得做paging转换，linear address又给page unit做paging转换。所以在Intel架构上既有segmentation又有paging，搞得比较乱。

paging怎么做的？就是我们前面讲的两级页表，10、10、12那样做。

在32位架构上出现了一个尴尬局面：硬件内存涨了，还是32位架构，但物理内存已经超过了4GB。这就引入了PAE（Physical Address Extension）——在32位架构上能去用超过4GB的物理内存。怎么做？把page table entry从32位改成64位，offset依然是12位，entry变成8字节。4K除以8等于512，所以两级都变成9位，再加上2位的page directory pointer table，变成了2、9、9、12的格式。因为是32位架构，虚拟地址空间还是4GB，但entry变成64位了，可以index 64GB的物理内存。后来64位架构一出来，PAE就被废了。

Linux在Intel上虽然支持segmentation，但没用16K个segment，只用了6个——kernel code、kernel data、user code、user data、TSS、default LDT。现在segment概念被进一步淡化，Linux用了两个mode：user mode（ring 3）和kernel mode（ring 0）。页表用的是五级页表，有PGD、PUD、PMD、PTE等等。

64位架构用的是39位或48位的地址，页面大小可以是4KB、2MB、1GB。如果是39位，virtual address format就是9+9+9+12。一个entry能index 4KB，那从第二级拿一个entry能index多大的page？9+12=21，2的21次方等于2MB。再下一级，三个9加12等于39，2的39次方等于1GB。

## 八、ARM架构

ARM比较晚，它可以选择最新的，不用都支持。它选的是只支持paging，不支持segmentation。ARM64也是只支持paging，不支持segmentation。

TLB是分层次的。在两级页表（10+10+12）中，如果第二级有一个TLB entry，能index 4KB；如果第一级有一个TLB entry，能index 4MB（10+12=22，2的22次方=4MB）。所以TLB分不同level，一般top level的TLB给几个就行了，几个就把内存地址空间覆盖完了，越往下level的TLB数量要更多。

## 九、分页与分段的起源

segmentation和paging哪个提出得早？很让人诧异——它们几乎是同时提出来的。segmentation是1961年在Burroughs B5000上用的，paging是1962年曼彻斯特大学Atlas电脑上用的。

那为什么早期的Intel支持segmentation不支持paging，现在又支持paging不支持segmentation了？难道真的是分久必合合久必分？原因是什么呢？不是内存大小的问题，而是硬件复杂性的问题。segmentation硬件很简单，搞个寄存器、搞个segment table就行了。paging你得搞TLB，TLB要求fully associative memory（可以并行查找），60年代谁会做这个？TLB不会做，page table walk也需要很多硬件，segmentation table不用walk，就是一个数组。所以paging的硬件设计复杂到早期的工艺不经济——造出来又慢又贵，没人用。后来随着硬件越来越好，大家才开始用更复杂的paging机制。

这就是硬件和软件协同发展的一个典范。大家觉得paging好，但你硬件造不出来，造出来也很贵没人买，所以退而求其次用segmentation。

## 十、课堂练习

假设页面大小是64KB，32位架构，virtual address format是什么？offset是多少位？64KB等于2的16次方，所以offset是16位。一个PTE entry多大？32位架构下是4字节。页面大小64KB除以4字节等于16K，16K等于2的14次方，所以中间这级是14位。前面剩下的就是2位（32-16-14=2）。

如果是64位架构，页面大小64KB，一个entry是8字节，64KB除以8字节等于8K，8K等于2的13次方，所以每一级是13位。48位的话，第一级就是48-16-13-13=6位。

这个得会算，一般内存这块期末考试会出一道比较难的题，跟实验结合起来，让你现场当CPU去walk页表。

## 十一、页面大小的权衡

我们最开始讲页面的时候问过：prefer大的page size还是小的page size？现在内存越来越大了，对那点internal fragmentation已经没那么敏感了。页面大了有什么好处？39位的VA，如果是4KB page size，格式是9+9+9+12，三级页表；如果是64KB page size，格式是？64KB page size offset 16位，entry 8字节，9+13+16？不对，应该是用39位算：offset 16位，entry 8字节，每级是13位，39-16=23，23不是13的倍数，所以需要调整。实际上64KB page size在39位下只需要两级页表。页表层数减少了，TLB压力就不一样，walk页表时也少walk一层。所以页面变大，页表层数减少，这是页面大小带来的好处。

## 十二、页表共享

最后讲一下page sharing。segmentation能不能share？可以，但share的粒度是整个segment。比如把base和limit设成一样，两个进程就指向同一段物理内存。但如果你想share一半，就比较weird。所以segmentation的share粒度是整个segment，不灵活。

paging呢？share的粒度是一个page。它可以很灵活地share一个4KB的页面。这里又涉及page size的问题——page越大，share越不灵活；page越小，share越灵活。在paging里面我们可以做到每个页面有独立的permission，非常灵活。
