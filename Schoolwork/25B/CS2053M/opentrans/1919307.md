接下来讲一下具体的寄存器设置。这一页展示的是sstatus寄存器的状态。我们重点介绍两个位：一个是SUM（Supervisor User Memory access）位。如果你把SUM设置成1，就代表S Mode下可以访问User Mode的那些页；如果SUM设置成0，那S Mode就无法访问User Mode的页。这个希望你们自己在实验中去设置一下。

还有一个是SPP（Supervisor Previous Privilege）位。如果SPP置成1，它可以让sret指令从S Mode返回到User Mode。主要就是这两个位的事情。

最后是PTE（Page Table Entry）。这次你们需要关注的是U位，也就是这里显示的第四位。如果U位是1，就代表这是一个User Mode下的页，User Mode可以访问；如果U位是0，就代表S Mode可以访问。这就是你们的页表项设置。

好，我们继续上课。刚刚讲了实验四的内容，现在我们来深入讲一下虚拟内存。

虚拟内存，其实就是我们用页表来构建的一系列新的概念。前面我们讲过了“逻辑内存”和“物理内存”，这两个概念我们不做很详细的区分。今天正好会讲到你们实验里面用到的内容——为什么我们会设计实验四？就是为了实现一个叫“Demand Paging”的概念。

在学虚拟内存之前，我们先复习一下前面两周讲的东西。从逻辑地址到物理地址的转换，我们定义过多少次？第一次是Paging的时候定义过一次，第二次是Page Table的时候定义过一次，第三次是单级页表的时候定义过一次，到两级页表的时候又定义过一次。后面还有三层、四层、五层的页表。所以这个“逻辑地址”和“物理地址”其实一直在变。我们为什么不用“固定地址”？因为固定地址不能变，所以我们定义了一个能变的，给它起了个名字叫“逻辑地址”。

后面我们学了内存分区（Memory Partitioning）的时候，其实隐含地涉及到了内存位置（Memory Locations）的概念——我们怎么样来找哪个分区能够放得下我们的进程。一般有Best Fit、First Fit和Worst Fit，选择题里经常会考到。

后面我们又学了Paging。Paging的基本思想是把进程切成Page，把物理内存切成Frame，让Page能够放到Frame里面——它们俩大小一样大，这样我们只需要在中间加一个映射就行了。这个映射的名字叫Page Table。我们重点学了页表的几种格式——总共学了几种呢？三种。Hierarchical Page Table（分层级页表）、Hashed Page Table（哈希页表）和Inverted Page Table（反置页表）。最常用的是哪一种？Hierarchical Page Table。为什么？因为它快——它本质上就是一个数组，你直接跳到对应的索引就能访问了。

然后我们学了一级页表怎么样把它转换成二级页表。从一级到二级学会了之后，从二级到三级、四级、五级就都比较简单了。从一级页表到两级页表为什么能省内存？有两个好处。第一，一级页表是一个数组，并且是物理连续的，如果第一项和最后一项有内容，中间的项也得给它分配空间。而使用两级页表，你只需要分配一个顶层页表（PGD），下层需要哪一个页目录项再去分配就行了，所以它能省内存。第二，因为二级页表把原来4MB的空间重新装成了一个Frame大小，可以放到任何地方，不需要物理连续——这就是两级页表相比一级页表的优势。这个知识点每年都会考，只要我出题一定会考。

之后我们又学了Swapping、TLB和MMU。MMU是Memory Management Unit，它的物理含义就是做从逻辑地址到物理地址的转换。这是我们前面两周学的内容。这个星期学什么呢？学一些可以基于页表和分页来构造的高级概念。第一个叫Demand Paging，第二个叫Copy-on-Write，第三个是怎么样管理物理帧。这个虽然叫Page Replacement，实际上就是物理帧用完了，我们怎么把页面踢出去。后面我们还会学Frame的分配、Thrashing以及Linux的内存管理。大概花一周或一个半周把这些内容讲完。

好，我们先看第一个——Demand Paging。什么叫Demand Paging？先学一下它的背景。代码需要放到内存里才能被执行，但是我们的程序很多时候不需要同时全部放到内存里面，放一部分就可以执行了。比如说有一些错误处理的代码、一些不常用的数据，一开始并不会被用到，你何必把它加载进来呢？所以很多时候我们需要的是“部分加载”——一个程序只加载一部分，而不是全部加载，这时候也能执行。你们有些人可能玩过大型游戏，电脑的内存可能只有1到2GB，但下的游戏有4GB大，电脑还能很流畅地把游戏运行起来，原因就是用了Demand Paging这个概念。

怎么样实现Demand Paging呢？其实就用到了我们学的Paging。这里面引入了一个新的概念叫Virtual Memory——虚拟内存。虚拟内存就是逻辑内存与物理内存的分离。我们用的Sv39页表，虚拟地址的索引能索引多大的内存？512GB。但你们有没有512GB的真实存储？没有。你们的手机物理内存一般也就4到12GB。所以物理内存并没有那么大，你只是“看上去”有这么大。我们把这种内存称为“逻辑内存”，也叫“虚拟内存”，因为它是虚的、假的。所有的进程、App都是在这个虚拟内存上面运行的，每个进程都以为自己有512GB的内存，实际上它们共享的是同一块物理内存——这就是引入虚拟内存的概念。

大家有什么问题没有？没什么问题的话，上面有些文字大家下去自己读一下，不是重点，重点我已经讲过了。虚拟内存可以远远大于物理内存——你只有4到12GB的物理内存，但是进程看到的虚拟内存远大于物理内存。这里我用红字标出来的大家要记住：虚拟内存其实不能叫“内存”，它只是一个地址范围，没有真正的存储空间。所有的存储都需要物理内存来给它做支撑。虚拟内存是空头支票，它只是标了个数字。甚至用48位地址，我可以说你有256TB的内存，但手机上实际只有4到12GB这么大。虚拟内存没有存储空间，只是标个数，物理存储需要Physical Memory来支撑。

在一个用户进程上——刚刚助教跟大家讲了实验四，这里面就有了User Space的概念。之前我们玩的都是内核空间，现在终于有了User Space。每一个User Space的进程，它的内存布局大概类似这样：最高地址是栈（Stack），中间会映射一些库（Library），下面是堆（Heap），最下面是代码段（Code Section）、数据段（Data Section）、BSS段等等。这是这个进程的虚拟地址空间，我们人为给它划分出来了一些区域。栈从高位地址往低位地址长，堆从低位地址往高位地址长，这样可以最大化地址空间的使用。栈和堆中间增长的时候，我们并没有真正地给它分配物理地址，一会儿我们会讲这个东西——它就叫Demand Paging，用Page Fault去处理。中间还映射了一些共享库，它们占物理地址，可以映射到这个进程里面，这样就实现了共享。

好，下面跟大家讲一个比较硬核的东西——Demand Paging。Demand Paging是什么意思呢？只有当一个页真正被需要的时候——被Demand的时候——它才会被映射进内存。这里我经常问大家一个问题：你们学C语言的时候分配一大块内存，用哪个函数？malloc。大家知道malloc是什么的缩写吗？Memory Allocate。你malloc一兆内存，你觉得操作系统会给你一兆物理内存吗？不会。那什么时候会给你呢？当他真正被需要的时候——当你去读或者写这块内存的时候。比如你有个buffer，当你真正去读这个buffer或者往里面写的时候，操作系统才会给你分配物理内存。这就是Demand Paging的含义。一开始你分配的就是一个空头支票。如果你去访问的时候——也就是你Demand的时候——操作系统说：“我再也不能延后了，现在就要用了，我必须得给了。”怎么给呢？如果这个Page是有效的（Valid），但没有在内存里，我就给它找个物理帧；如果这个Page是Invalid的，就直接报段错误（Segmentation Fault）。

这里面问几个问题：谁引起的Page Fault？是用户程序——它真正去访问这个地址的时候发现没给，所以引起了Page Fault。谁触发的Page Fault？MMU。谁处理的Page Fault？操作系统。

我们一起来看一个图，这个图把刚刚说的整个过程给具象化了。这是国外一个大神画的图，版权归他。我觉得这个图很好，就拿来用了。蓝色的块是虚拟内存，绿色的块是物理帧（Physical Frame）。用户程序先调用brk来增长它的堆。brk是干嘛的？你调了malloc，操作系统会把malloc编译成brk这个系统调用。brk帮你增大堆。假设虚拟地址空间只有两个页、8KB这么大，现在它想要更多，想要额外8KB。操作系统就给了它8KB，现在总共16KB了。注意，操作系统什么都没做——只是把账本里的数字从8改成了16，新的页没有物理内存支撑。程序信以为真，就去访问这个内存了。

当程序真正去访问这个内存的时候，操作系统说：“坏了，我只是给它打了空头支票，它还真当了。”然后操作系统把User Space的程序暂停，触发一个Page Fault。Page Fault处理程序找一个空闲的物理帧，把它映射到这个页上，然后恢复程序的执行。程序没有意识到背后发生了什么，它以为就是正常访问内存，好像比平时慢一点，但还是访问到了。这就是整个Demand Paging的处理流程。最关键的一步叫Page Fault，它会触发一个中断（Interrupt），这种特殊的中断就叫Page Fault。

这个流程大家应该都比较直观。MMU怎么知道这是一个Page Fault呢？页表里有一位叫Valid/Invalid位。在RISC-V的Sv39架构里，这个位叫什么？叫V位。它如果是1，就说明有映射；如果是0，就没映射。这个V位谁来设置？操作系统。实验里你们要自己设置——映射了就设成1，没映射就设成0。当User Space程序去访问一个地址，MMU去查页表，看到V位是0，就知道这一页只有虚拟内存，背后没有物理内存支撑，于是就触发了一个Page Fault。谁去处理Page Fault？还是操作系统。

这是一个详细的处理流程。第一步，先检查VMA（Virtual Memory Area），判断这个Page Fault是不是合法的。VMA就是操作系统的账本。操作系统在MMU触发Page Fault之后，过来先看看——哦，你访问的地址落在我这16KB的堆区域里面了，这是一个正常的访问。然后操作系统就去找一个物理帧映射上。如果访问的地址在外面呢？也会触发Page Fault，但这种就是非法访问，操作系统会报段错误（Segmentation Fault）。VMA我们一会儿会细讲，你就记住它是一个闸门——地址在VMA里面，就找物理帧映射；地址在VMA外面，就报段错误。

你们的Lab4大概就是实现这样一个步骤。说起来比较简单，就是你们Lab4当时设计的时候，主要就是为了让你们实现Demand Paging这个很聪明、很伟大的概念。

好，回到我们的老朋友——ELF文件。ELF是C程序被编译成的可执行文件，它可以映射到这些区域。另外还有两块动态内存，一个是堆（Heap），一个是栈（Stack）。这是整个进程的内存布局，分成这么多区域，相应的VMA也有这么多区域。代码段有一个VMA映射，数据段有一个VMA，BSS段有一个VMA，共享库有一些段映射，栈也有一个VMA。问题来了：为什么要分成这么多区域？能不能用一个VMA把整个进程的所有内存都映射起来？答案是不能的。大家看一下这些VMA里面标注的文字有什么不一样？权限不一样。代码段是只读可执行（R-X），不能写；数据段是可读可写（RW-），不能执行。权限都不一样，所以不能用同一个VMA。另外，地址空间不连续，中间有洞。VMA就是把连续的区域放一块。一个进程可以有几十、几百甚至上千个VMA。

VMA在Linux内核里定义为一个结构体，里面有两个关键的字段——vm_start和vm_end，还有权限等等。就像刚刚我们把堆增大了8KB，操作系统做了一个什么事呢？就是把堆对应的VMA的vm_end加了8KB——就做了这么简单的一件事。

我们来总结一下整个流程。第一步，程序调用brk系统调用。第二步，操作系统把堆的VMA增大，新的页没有映射物理内存。第三步，程序去访问新的内存，MMU触发Page Fault。第四步，操作系统处理Page Fault——先遍历所有VMA，看这个地址是不是落在某个VMA里面。如果落到外面，就触发段错误；如果落在里面，就去处理Page Fault。操作系统找一个空闲物理帧映射上，然后恢复程序执行。

这里面有一个效率问题：如果VMA有成百上千个，线性查找效率不高。Linux用什么数据结构来组织VMA？红黑树（Red-Black Tree）。Linux内核里面大部分数据结构都是数组，链表已经顶天了，但为了提升查找效率，这里用了红黑树。每一次Page Fault都要去查这个红黑树，所以它的性能至关重要。

第二个问题：除了堆，其他区域会不会发生Page Fault？会的。数据段和代码段也会发生Page Fault。有一种极致的省内存模式叫Pure Demand Paging——进程开始执行的时候，内存里一个页都没有。第一条指令就会触发Page Fault。ELF执行的第一条指令在哪里？对于静态链接，是_start；对于动态链接，会先执行loader，然后跳到_start。现在内存里什么都没放，突然要跳到_start执行，就会触发Page Fault。这个Page Fault和堆上的Page Fault有什么不一样？堆是匿名的（Anonymous）——没有文件支撑，随便拿一个Frame映射就行了。而代码段有文件支撑，操作系统需要先把文件的这一页从磁盘读到内存，然后再映射。所以代码段的Page Fault处理更复杂——要先从磁盘读，等读完之后再把Frame映射到页表里，然后重启这条指令的执行。

我们来看一个可执行文件的例子。假设代码段在页0和页2，数据段在页5。页表该怎么画？哪些页表项是Valid的，哪些是Invalid的？有映射的就是Valid（V=1），没映射的就是Invalid（V=0）。权限也不一样——代码段是R-X，数据段是RW-。程序执行的时候，先执行A，里面有个函数跳到C，C就会被加载进来。如果A又执行到B，B还在磁盘上，就会触发Page Fault，操作系统把B从磁盘读到内存，映射上，然后继续执行。这就是Demand Paging——按需加载。

整个处理过程分10个步骤。第一步，陷入操作系统。第二步，保存用户寄存器和进程状态。第三步，判断中断是Page Fault，检查地址是否合法。第四步，找空闲物理帧。第五步，确定页在磁盘上的位置，发起读请求。第六步，等待磁盘I/O期间，把CPU让给其他进程。第七步，收到磁盘中断，标记进程为就绪。第八步，等待CPU分配给这个进程，做上下文切换。第九步，修正页表，映射新的帧。第十步，返回用户态，恢复寄存器和状态，继续执行被中断的指令。

这是最坏的情况——页面在磁盘上。如果只是堆或栈上的Page Fault，没有磁盘I/O，那第5到第8步都可以省略，直接从第4步到第9、10步就行了。
