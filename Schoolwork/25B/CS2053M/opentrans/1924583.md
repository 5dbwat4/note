# 虚拟内存与Linux内核

## 一、课程导入与整体安排

好，我们开始上课。我们的软件部分还有四个周的课程，今天我们把Memory这块争取讲完，这个周一定要把它给讲完。然后我们就开始讲I/O，然后会跟大家讲一些文件系统。你们的实验，软件部分已经全部都能做了——内存那块已经全部都能做，fork也讲过了。这次课我们把Working Memory这一块跟大家讲完，还是用的这一套来自工业界专家的幻灯片。

上次课讲过的内容我们不详细讲了，如果你们不清楚的，把课程视频翻出来再看一遍。上次课讲到，在虚拟内存中，CPU、进程还有内核，用的都是虚拟内存。虚拟内存要做划分，把User Space和Kernel Space划分出来。在32位场上，虚拟地址空间总共是4GB，我们把它划到C0000000这个地方。C是1100，所以用这个划分，我们把高的1GB给内核，低的3GB给用户空间，用户空间看到的是低位的这3GB。

## 二、内核与用户空间的地址划分

这里面有一个问题：为什么给操作系统的虚拟地址空间反而少，给User Space的虚拟地址空间多？为什么Kernel的小于User的？第二个问题，我们跟系统二的部分串起来——在系统二我们学的是把CPU分成了User和Kernel，现在我们又把内存分成了User和Kernel。CPU是怎么来划分User和Kernel的？Memory又是怎么来划分的？

先回答第一个问题，为什么给Kernel分配的空间比给User分配的空间小？操作系统不应该给自己更多吗？为什么虚拟地址只给了这么一小段？想一想。为什么给User Space的多？因为User Space跑的是进程。回顾操作系统最初的概念，操作系统是管资源的，它不能监守自盗——不能说管资源就把所有资源都给自己。它是给进程管资源的，而进程运行在User Space。最重要的资源、最多的资源要交给User Space的进程来用，而不是操作系统自己用。理论上，给操作系统分配的虚拟地址空间应该是越小越好。这样的话，你把更多的虚拟地址范围给User Space，虽然这个空间是假的，用户空间的进程不知道，它觉得你给得越多越好。所以我们设计了3GB/1GB的划分，当然你也可以用2GB/2GB的划分，但这种划分一般不多。

强调一点：虚拟地址背后是没有映射内存的，什么时候映射？是你实现Demand Paging、真正去访问的时候才会映射。所以用户空间虽然看到3GB，但3GB都是假象，它以为有3GB，实际上没有这么多。

## 三、进程与页表的对应关系

上次课讲了进程跟页表之间的对应关系——一对一，一个进程就有一个页表。这就是之前那张图里面代码段、数据段什么的，都是通过页表来索引的。系统二学过Context Switch，当时没有让你们随意取页表。其实这里面的问题是：当你切换进程的时候，页表需要不需要切换？需要切换。如果需要切换，具体的操作是什么？上次课讲了一个关键数据结构叫mm_struct，mm_struct里边哪一个成员指向页表？是PGD。切换页表很简单，就是把PGD的地址放到Page Table Base Register里面，即ASID或者TTBR。所以我们有switch_mm，这是Context Switch里面去switch mm_struct。

Linux内核里头，RISC-V架构的switch_mm代码总共就这么长。哪一行做了把PGD写到SATRP寄存器？六十二行。它要把新进来的进程的mm_struct里拿到PGD——上次讲过，这里面存的是虚拟地址。有同学问为什么存虚拟地址，因为可能设计上这个地方存物理地址也行，但它存的是虚拟地址，所以要先做虚拟地址到物理帧号的转换，然后返回值写到SATRP里面。每一次进程切换都要切换页表。

## 四、CPU与内存如何划分User/Kernel Mode

CPU是怎么来划分User和Kernel Mode的？通过一个Mode Bit，就是那个MSR寄存器的Mode Bit。它把指令分成了Privileged指令和Unprivileged指令，Unprivileged指令能在User Mode跑，Privileged指令能在Kernel Mode跑。

内存怎么样来做切分？通过虚拟地址切分：你访问大于C0000000的地址，我们就知道你访问的是内核地址；访问小于C0000000的，就是User Space地址。如果一个User Space的页表里面竟然映射的有Kernel Space的东西，那么你在User Mode来访问Kernel的内存，直接就给你干掉了，不让你访问。

## 五、64位地址空间划分

这是对于32位，地址空间有限。对于64位呢？64位地址空间太大了，根本用不完，所以我们在头上一头截了一点给User Space，高位截了一点给Kernel Space。具体截多少呢？如果是39位的地址空间，高的地方截512GB，低的地方也截512GB。如果是48位的地址空间，高的地方截256TB，低的地方也截256TB。因为它太大了，明显用不完，截太多也没有用，已经远远超过物理内存的大小了。512GB、256TB，很少有设备内存大于256TB，完全可以用这种配置来覆盖几乎所有的设备了。还有57位的配置，那是给超级计算机使用的，再大也没有意义了。Linux内核前几年默认用的是39位，这几年随着AI越来越吃内存，就换成48位了。

## 六、地址范围的计算

48位的User Space内存范围是多少？从0x0到2的48次方减1，也就是0x0000000000000000到0x0000FFFFFFFFFFFF。前面有16个十六进制位没有用，因为总共64位，其中有16位是前缀。这个范围是从0开始，到48个二进制位全1结束。如果是39位的，前面有64减39等于25个1，也就是6个F再加上一位——从0xFFFFFF8000000000开始。如果你看System.map，从地址就能看出系统编的是多少位的虚拟地址。比如地址以FFFF开头，后面跟着F，F前面还有C，那这是39位的，因为39位前面的地址都被保留了，有特殊用处，内核给自己映射就是从C开始的。如果要看配置文件，.config里面有VA_BITS=39或者=48这样的配置。

## 七、物理内存到内核地址空间的映射

虚拟地址空间需不需要把所有的内存都映射到内核里面？假如系统有8GB内存，用的是39位的VA，内核地址空间映射范围是256GB，物理内存只有8GB。这8GB需不需要全部映射到内核地址空间？需要。内核虽然不能全用——它不能监守自盗，8GB应该大部分都被微信、支付宝这些用户进程用了——但是内核要管理内存，它需要知道有这些物理内存存在。如果内存在内核里面没有地址，操作系统想把它分配给User Space也没法分配，因为不知道地址是什么。所以每一块物理内存都需要有一个内核地址。但大家注意：有内核地址并不代表内核使用，内核只是管理它，内核有可能把它映射给User Space。所以它都需要有一个内核地址。

这个东西在系统启动的时候会有一个页表叫init_mapping，那个映射是一对一的线性映射。启动完之后，为了管理物理内存，它要把所有物理内存全部映射到内核的虚拟地址空间，这个叫线性映射，是一对一的连续映射。这就是内核怎么样去管物理内存——都要把物理内存映射到它的虚拟地址空间。

在64位里面，虚拟地址空间一般比物理地址空间大得多，所以完全映射进来没问题。在32位上，如果物理内存小于896MB，直接全部映射进来。如果物理内存小于896MB，就把它映射到内核最开始的位置。

## 八、线性映射与地址转换

线性映射的定义是：物理地址与虚拟地址之间有一个固定的偏移量，叫作PAGE_OFFSET。物理地址0映射到虚拟地址C0000000，物理地址0加1映射到虚拟地址C0000000加1，它们之间的偏移就是PAGE_OFFSET。这是Linux内核里的一个宏。

如果你有一个虚拟地址，想算出物理地址，减去一个PAGE_OFFSET就行了。反过来，如果有一个物理地址，想算出虚拟地址，加上PAGE_OFFSET。内核为什么要这样做？就是方便管理——每一次做物理地址到虚拟地址的转换不需要走页表，直接做一个线性的映射就可以了。在Linux内核源代码里可以直接看到这两个宏的定义。你如果去搜__pa和__va，就能看到它们的定义。线性映射——只有线性映射才有这样的转换。如果不是线性映射，而是走页表的，就不能这样减。

## 九、三种地址类型

这种线性映射的地址有一个名字叫Kernel Logical Address。你很少看到Kernel Logical Address和Kernel Virtual Address的区别，但万一有人讲到这个东西，你得大概知道。线性映射就是64位里全部映射进去，32位里物理内存小于896MB的直接映射进去。

那如果物理内存比虚拟地址空间大怎么办？比如39位系统，老板给配了2TB的内存，比地址空间还大。或者说32位系统上内存大于896MB怎么办？这是当年一个现实的问题——当时架构还是32位，但手机内存堆到了4GB、8GB。

早期Linux是怎么做的？物理内存过大，有一块做线性映射——底下的896MB做线性映射，1:1映射。上面还有128MB的虚拟地址空间，这128MB反复使用。先把物理内存分成两块，一块是896MB的叫Normal Memory（常规内存），大于896MB的叫High Memory（高端内存）。从High Memory里先截一块，映射到那128MB的虚拟地址空间里。管理完之后，把那块虚拟地址空间释放掉，再映射下一块物理内存。这128MB的地址空间被重复使用，保证能把所有物理内存都用上。这块内存叫Kernel Virtual Memory。当时Linux有一句名言：做这样设计的时候，没想到这个东西会被用上，因为觉得架构和硬件会一起发展，结果硬件堆得比OS架构发展得快了。大概2010年到2013年，几乎所有的智能手机都用的是这种架构。后来64位架构出来，这块东西大家再也不用care了。

## 十、三种地址的详细分类

在Linux Device Drivers第三版第15章里，地址被分成三类：User Virtual Address、Kernel Virtual Address、Kernel Logical Address。这张图非常关键——中间是物理内存，左边是User Space，右边是Kernel Space。Kernel Space里又分成Kernel Logical Address和Kernel Virtual Address。

内存大于896MB的称为High Memory，小于等于896MB的称为Low Memory。User Space有没有线性映射？没有。User Space的箭头画得乱七八糟，没有线性映射。如果拿到一个User Space的虚拟地址想转成物理地址，只能去Walk Page Table。Kernel映射Low Memory的时候是线性的——物理地址和虚拟地址之间有固定偏移。Kernel映射High Memory的时候是非线性的，箭头可以乱指。

如果Kernel Logical Address里的虚拟地址想找对应的物理地址，人可以减PAGE_OFFSET，但硬件MMU不知道减offset，它只能走页表。所以虽然是线性映射，也得有相应的页表映射。人一般都减offset，而MMU硬件一般去Walk Page Table，两种方式都可以。Walk Page Table是拿到地址转换最底层的方式。

Kernel Virtual Address如果想转换成物理地址，只能去Walk Page Table，没办法减去offset。

## 十一、连续映射与不连续映射

哪一段内存是连续映射的——虚拟地址连续，物理地址也连续？Low Memory那一段，因为它是线性映射。User Space里虚拟地址连续，但物理地址不连续。一个进程里面的虚拟地址一般是连续的，但物理帧不连续，取决于分配到哪一个帧。High Memory在Kernel Virtual Address区域里，虚拟地址连续，物理地址不连续。

## 十二、页表与进程隔离

每一个进程有它自己的页表，这就是一对一的关系。两个进程访问同样的虚拟地址，访问的内容一样不一样？不一样。通过什么让它们变得不一样？通过页表。进程二有没有可能访问到进程一的内存？如果不共享，根本没有可能——页表把控着，你访问的虚拟地址映射到不同的物理帧，所以访问不到别人的内存。页表把控着内存的映射，它是一个特别关键的数据结构。这个数据结构应该被放到内核态还是用户态？内核态。如果页表被放到用户态，用户进程就能随意修改它，说我想访问哪块内存，直接把它映射到我的页表里就行，操作系统就失去了对内存的管理和控制能力。操作系统通过把页表放到内核态来实现对进程的管理，也通过页表来实现不同进程内存的隔离。

如果想实现Shared Memory，也很简单——把同一个物理帧映射到两个进程的页表里就行了。你们的系统里像libc几乎被映射到每一个进程里面，所以肯定是共享的。物理内存里只用存一份libc，大概两兆，然后映射到各个进程里。如果不做共享映射，每个进程都要自己的两兆libc，1000个进程就到2GB了。所以Shared Memory能够很大程度上来省内存。Shared Memory可以通过mmap系统调用来实现。

## 十三、MMU的工作原理

MMU是Memory Management Unit，是一个硬件组件，负责从虚拟地址转向物理地址。MMU会不会去减那个PAGE_OFFSET？不会。MMU很傻，它不知道线性映射，也不知道物理地址是不是连续，它只管去Walk Page Table。如果页表里有映射，就把地址给你；如果没有映射，就触发Page Fault；如果是Invalid的，也触发一个Fault。然后操作系统去查VMA，判断到底是个Page Fault还是Segment Fault。MMU操作的基本单元叫做Page。Page的大小随架构变化，最常用的是4K，ARM64可以用4K或64K，x86一般是4K。

一个Page是虚拟地址空间里划分出来的单位，物理地址里称为Frame。Page是抽象的，只是一个大小范围，并没有存储；Frame需要真实的硬件存储，需要真实的内存。内核里的PFN缩写是Physical Frame Number，但很多人不严谨地叫Physical Page Number，严谨的叫法是Physical Frame Number。

## 十四、Lazy Allocation与Page Fault

当一个进程去访问一块没有被映射的内存时，会出现Page Fault。操作系统来处理这个Fault，判断是不是Page Fault，如果是就分配物理内存；如果不是，是Segment Fault，直接把进程干掉。

操作系统访问用户空间的内存请求时不是立即响应的，它只等进程真正去用这些Page的时候才去映射。因为操作系统知道物理内存一般都会over-commit，用户空间会请求很多内存但并不一定会全用，有很多冗余。这个优化叫Lazy Allocation，是为了性能。进程请求更多内存时，内核只是记录一下，告诉他“你有了”，但不会真正去映射。当新内存被访问时，MMU会生成Page Fault，在Page Fault Handler里面操作系统来决定这个映射是不是Valid，如果是Valid就更新页表，找一个Frame映射上，然后回到User Space继续执行。

VMA在这里很关键，用来决定一个地址是不是Valid。第一步是请求，操作系统只记录了一下，虚拟地址空间增长——从8K涨到16K，但这些都是假的。当你真正去访问时，触发Page Fault，操作系统在Handler里给你做真正的映射。VMA用红黑树来查。在Linux里面，User Space是不知道Page Fault发生了的，它只知道访问内存好像很慢。本质上所有东西都是操作系统帮它做的。

如果一个进程是时间敏感的，数据可以被Prefault进来，就是提前映射好。有mlock和mlockall这样的系统调用可以做Prefaulting。

## 十五、Swap与总结

页表里的条目在TLB中数量很少，所以要把它们存到内存里，这个数据结构就叫Page Table。

当内存不够用时，需要用Swap来换出一些Frame到磁盘，释放RAM。这时把一个Frame写到磁盘上，把对应的物理Frame回收，给其他进程使用。

最后总结一下：内存管理讲了三大块——Partition、Segmentation、Paging。Paging跟Segmentation几乎是同时发明的，但Paging相比Segmentation的优势是什么？Segmentation很难实现Demand Paging，而基于Paging衍生出了很多高级用法，比如Demand Paging、Lazy Allocation，这些都是Segmentation单独实现不了的。
