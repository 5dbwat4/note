## 二、大容量存储结构概述

好，我们这一次就讲mass storage——这个mass就是大量存储。我们的mass storage主要学什么呢？第一个是这个mass的结构，主要是磁盘的结构；磁盘呢还有调度，还有磁盘的管理，还有swap space的管理；最后我们会学RAID，就是一个冗余——我们怎么样用磁盘做冗余。

大家对磁盘有什么印象？我们现在好多同学的电脑上应该也是磁盘，就是hard disk。当然也有一些同学的笔记本电脑用的是SSD，我们这次课就学它们的区别。

磁盘是我们现在绕不过的一个概念。十年前几乎所有的笔记本电脑用的都是磁盘，现在SSD逐渐替换了一大部分电脑，但当你有大规模存储的时候，比如说你想有十个T的存储，你可能还是磁盘更划算一点，SSD费用更高一些。

磁盘最常见。hard disk一般都叫magnetic disk，因为它里面用的是磁性材料做存储。它附着到这个I/O bus上，这个磁盘它会转。磁盘的盘片有platter，排列从0.85英寸到14英寸都有，现在用的是3.5英寸、2.5英寸、1.8英寸的。容量从30GB到3TB的现在都有，这是现在的磁盘。

我不知道有多少同学拆过你电脑的磁盘，没拆过的举下手——大家都拆过啊？有少数的同学拆过，一般是多大？现在磁盘大多大就大概这么大。

大家猜以前的磁盘有多大？这是1956年的磁盘，这个磁盘只有5兆，5兆字节，每个字节只有7个bit，这么大，跟个柜子一样大。这是195几年的磁盘，磁盘的发展史几乎就是IBM公司的发展史，从最初的工程师发现有磁性材料，到后来出了很多商业产品，每一个大型机都配了很多个这样的磁盘。这个磁盘容积很小，只有5兆——现在你们买个U盘都在128G了，这只有5兆，但是体积竟然这么大。


## 三、磁盘结构

磁盘的结构是什么样呢？你可以把它理解为一维的一个数组，里面分成了很多logical block。这个logical block是最小的传输单元，logical block被映射成了sector（扇区），还有track（磁道），不同磁盘的扇区叫cylinder（柱面）。

磁盘的结构大概长这样：这个磁盘有三个盘片，这里面有很多术语，大家一一把它认一下，有可能再过20年这个东西就成古董了，你们就再也不用学磁盘了。但是现在你们还在学磁盘。

这个是spindle，是转动的马达。这三个东西都是磁头，叫head。每一个磁头伸到这个盘片上，它的移动是这样移动的，不是说直接这样，它是摆动。盘片在旋转，磁头摆动。这中间一环叫track，盘片叫platter，多个盘片形成了上中下，这个叫cylinder。

每次讲到这儿呢，我都会讲这个概念：磁头只能在这个地方移动，这个盘子就在转。所以这里面有个问题：现在磁头在最外道的track上，如果想读15这个扇区，盘片和磁头该怎么配合？盘片转到15，磁头往里面移，移到这个地方，然后盘面开始转。


## 四、磁盘性能与访问时间

好，我们刚刚也提过那个问题，就是磁头在9这个track上，在外道的track，如果他要读15，这个时候磁头要挪，盘片也要转。磁盘挪和盘面转这两个动作都需要时间，所以引入了两个delay。一个叫positioning time，就是把磁头挪到想要的sector上。

positioning time包含seek time和rotational latency。seek time是什么意思呢？就是把磁头挪到target的cylinder上，挪到这个柱面。如果它是一个盘面，它就是一个track；如果两个盘面就有上下一个柱面，你得先挪到这个柱面上，然后你再去转，到它要的那个sector。

另外还有一个叫transfer time，这个也需要时间，主要取决于transfer rate，理论上能达到6Gb每秒，但实际上达不到，实际上可能能达到1Gb每秒。所有的latency几乎就是这些东西加起来。

大家觉得现在磁盘速率怎么样？快还是慢？确实很慢。为什么慢？就是因为它里面有机械装置——有个马达，马达能转多快？第二个你还得来回挪磁头。这两个其实都是很费时间的。这两个机械装置就导致磁盘又慢又不可靠，容易损坏。所以现在磁盘已经大规模被SSD取代了，SSD是用电路来实现的。

seek time一般是从3毫秒到12毫秒，9毫秒是average。这个时候我们就能把整个的时间画出来了。磁盘在最下面竟然需要9毫秒。大家还记得内存的访存时间吗？CPU假设1G赫兹，CPU的时间是多少？1纳秒。内存一般可能是200纳秒，所以CPU和内存之间是200倍。磁盘和内存之间呢？这是10的-9次方，这是10的-3次方，所以他们俩之间是10的6次方倍——一百万倍。所以磁盘绝对是最大的瓶颈。

现在你的电脑一般换成SSD之后，你会发现电脑速率飞起来了，以前可不行。

对于磁盘，我们要学会算average access time。I/O time就等于average access time加上数据传输的时间加上controller overhead。average access time就等于average seek time加average latency。

假设我们要传一个4KB的block，在一个7200转的磁盘上，average seek time是5毫秒，传输速率1Gb每秒，有0.1毫秒的controller overhead。最后的时间加起来大概是10毫秒。就算我们把后面的忽略掉，这两个加起来也将近10毫秒，还是挺慢的。

大家算一下中间这个传输时间是多少？4KB除以1Gb每秒，4K bytes等于32K bits，除以1G等于30×10的-6次方，大概等于30微秒，也就是0.03毫秒。所以磁盘最多的overhead花在哪儿了？花在average access time，就是seek time和rotational latency。传输只有0.03毫秒，慢就慢在这两个上。

这两个都是什么时间？就是你挪磁头跟转盘片的时间。这两个能不能提升？很难提升，因为不想把磁头移动得太快——移动太快容易刮到磁盘。所以整个磁盘慢就慢在磁头移得太慢了。这也是你们跟小白的区别，下次你跟别人讲磁盘为什么慢，就因为它磁头挪得太慢了。


## 五、磁盘调度（Disk Scheduling）

磁盘很慢，内存一直给它发request，但磁盘受制于比较慢，很难响应，那怎么办呢？我们就有了disk scheduling。我们前面都学过哪些scheduling？CPU有CPU scheduling，内存有page replacement，磁盘也有disk scheduling。我们这三大资源操作系统都有管，都给了scheduling算法。

disk scheduling就是为了减少磁头移动的距离，增加传输的带宽。有很多pending的disk request——这个进程让我读这个，那个进程让我读那个，这么多request都来了，磁盘太慢了，不知道先serve哪一个了，所以就给它一个scheduling algorithm。

最初是操作系统来管，在很早很早之前；现在硬件逐渐能做smart了，有电路板了，电路板上跑了一个固件，就能帮它安排队列做管理。所以现在操作系统就没有设计这个了，像我当时读研究生的时候，我们那个实验我还编了一个磁盘调度。

讲到磁盘调度，你们能想到的第一种调度算法是什么？对的，就是first come first serve。以后你们做系统或者做research的时候，就先出一个最naive的版本，最傻的版本就是first come first serve。

我们一般会给一个request的queue，这些都是柱面的number，比如说98、183、37、122、14、124、65、67。柱面的范围是从0到199，磁头初始在53。下面就是让磁头移动距离最短。

first come first serve就是：现在磁头在53号柱面，先serve98，移动距离就是98减53等于45；serve完98再下一个是183，从98移到183；然后再从183移到37，从37移到122，从122移到14，从14移到124，从124移到65，从67。明显的看到磁头有来回移动，这肯定不是最优的算法。

评判一个算法是不是最优，会把它移动的所有距离记录下来。大家算一下first come first serve磁头移动的总距离是多大——等于640。这个期末考试会考大家一道题，是大题，也会考选择题，得会算。很简单，都是1000以内的加减法，不用计算器。

first come first serve明显不太好。根据前面CPU scheduling和page replacement，下一个应该是一个optimal的。该怎么样优化？这个特别难，因为没有一个optimal的算法。你可以想想怎么让它尽量优化一点，别让它来回走。

找一个更近一点的——shortest seek time first。这个东西就是我找离我这个磁头最近的request，我就去serve它。反正我移动的路上早晚都要碰到，比如说我移动到98的过程中肯定要经过56和67，那我索性先把这两个给serve了呗。

它的基本idea就是select the request with minimal seek time from current head position。但是跟我们传统的相比，shortest seek time first竟然不是最优的。你会发现它其实有问题：到53、65、67之后，下一个又走到39、走到14，又回来了。理论上我往这边走一下、往那边走一下就能把所有request都serve完。

提前跟大家剧透一下：disk scheduling竟然没有一个最优的。第二个，因为磁盘比较慢，所以磁盘一般会积累好多个disk request，它已经知道了这个list，然后再去里面慢慢挑哪一个是我要serve的下一个，这是它跟内存访问不一样的点。内存访问我们不能read the future，不知道下一个会访问哪个page，但现在磁盘的request你可以知道——在这个时间段它积累了好多个磁盘request还没有响应，这些都是它要去响应的，它可能下一个时间会来更多。

shortest seek time first也不是最优的。大家算一下这个距离是多少？这个很好算，就剪三下就出来了，等于236。

那有没有一个更简单一点的方法？早期的计算机大神就说：我能不能从一头到另一头，然后再拐过来，在这个移动的过程中把所有的扇区都走一遍，也不关心你是多少了，就这样来回走就行了。这种算法叫SCAN，因为它有点像电梯下去接人、上来也接人，所以叫elevator algorithm。磁头从一端移动到另一端，过程中把中间所有扇区的request都serve掉，到头了之后movement被reverse，从反方向继续，就这样来回serve。

大家算一下SCAN移动的距离——也是236。这个时候你就发现上面那个其实并不optimal。

这个算法能改进，怎么改进？先说一下它的变种：C-SCAN（circular SCAN）。就是磁头从一端移动到另一端，一到上头之后赶紧把它挪回到最低端，再从最低端移动。在从最高挪回到最低端的过程中不serve任何request，这就是单向的circular SCAN。

SCAN和C-SCAN有没有优化的方法？衍生了LOOK和C-LOOK。LOOK就是不移动到顶端，只移动到最高的那个要serve的cylinder；最低的地方也不移动到0，移动到最低的要serve的request。这样又能省一些磁头移动的时间。

那下一个问题就是：既然有这些算法，分别适合什么样的场景？什么样的场景该用SCAN和C-SCAN，什么样的场景该用shortest seek time first？

如果系统负载很大，一直有人大量request，I/O不停地读写各种柱面，这个时候该用哪一种？应该用LOOK和C-LOOK，它们有better performance for systems that have heavy I/O load。你负载很重，我也不管你谁近谁远，反正就整个走一遍，把路上的request都给serve了。第二个问题，light load的时候我们用谁？shortest seek time first。平常电脑用硬盘的时候一般会用shortest seek time first。

用SCAN已经是236了，比first come first serve的640少了很多，如果用LOOK、C-LOOK还能进一步减少。所以scheduling algorithm还是有用的。

除了磁盘调度，文件的分配、文件头和metadata以及文件内容放置的位置，也会决定磁盘的performance。文件系统一般会去优化，增加局部性，让你读一个block的时候把文件所有东西都读到。


## 六、固态硬盘（SSD）

下面我们学一下SSD，固态硬盘。固态硬盘比较专业的说法是non-volatile memory，就是非易失内存——内存掉电了数据还在。它没有机械组件，所以比磁盘更可靠。缺点是什么呢？就是贵。另外还有一个意想不到的缺点：它的寿命有可能比磁盘还短。它里面的cell擦写次数是有限的，比如你反复写大量文件、删大量文件，过不几天SSD可能就废了。

它贵，速率快得多，容量比较小。它嫌bus总线太慢了，会直接连到更快速的总线上。没有moving parts，所以没有seek time也没有rotational latency。对于固态硬盘，大家觉得用哪一种scheduling比较好？没有moving parts了，来了就没有远近了，所以就按first come first serve处理就行了。

固态硬盘也叫non-volatile memory，它一般做成跟盘片一样，没有moving parts，跟磁盘完全结构不一样。里面分成了不同的page。你可以去写、读，写之前要先擦掉。擦的次数多了，里面存的电就分不清到底是正还是负了，存的信息就废了，这一块就存不进信息了。所以它一般有一个固定的擦写次数，比如100K，大概10万次。

它的寿命用DWPD（drive writes per day）来度量，就是每天可以写这个drive几次。比如说一个1TB的NAND drive，rating是5 DWPD，就相当于每天写5TB的数据，在warranty时间内依然不会失效。

非易失内存还有自己的scheduling algorithm，是为了防止一块被反复擦写——如果一个地方被反复擦写，就把它挪到其他地方，最后让整个磁盘里擦写的次数尽量均衡，这叫wear leveling。


## 七、其他存储介质

除了HDD（hard disk drive）和SSD（固态硬盘），大家还能想到其他大规模存储的介质吗？磁带。磁带在日常生活中已经不常用了，在哪些地方用呢？在银行、在阿里云这样的大公司做备份——几百T的东西写到磁带里，磁带特别大。

磁带的优点是便宜，量大管饱。缺点是什么？很慢很慢，而且不支持random access（随机访问）。比如说你在set 0上，想访问最后一个set，它真的得把磁带整个走一遍。对于磁盘来说，可以挪磁头就行了，不用把前面的走一遍；对于固态硬盘更不存在问题，通过电路访问，不管哪里的时间都一样。但对于磁带，问题就是特别慢，移动很慢。好处就是特别便宜，量大管饱，所以一般用来做备份，相对来说permanent的存储——做个备份可能放个十年八年。

这三种介质讲完了：磁盘、固态硬盘、磁带。以后再想想以后过个多少年还会有更好的存储介质。

作为professional，不能就只知道这么几点。有同学见过软盘没有？当年一个软盘可能大概几兆到几十兆，有个口可以拉开，里面能摸到一个软软的盘面。因为特别软，转速很低，传输速率也很低。当年的Windows什么的都是通过软盘来卖的。在软盘之前还有纸带打孔——一摞打孔的东西，代表一些含义。


## 八、磁盘管理（Disk Management）

我们怎么样从硬件level来管这些存储？第一个，磁盘的management要做物理格式化。格式化的意思就是把磁盘画一画、分一分，相当于画好方格，知道哪个地方存什么东西。格式化完了之后还要在上面造一个文件系统，把磁盘划分成区，知道每一个有多大、有多少区域，方便往里面写。

操作系统会在磁盘上记录一些数据结构，比如说记住partition。另外一个也会做logical formatting，在上面造一个文件系统。

有多少同学格式化过硬盘或者U盘？大多数同学都没有格式化过硬盘？这真让我惊诧。格式化硬盘的时候，一般会在上面构建一个文件系统。

第二个问题：如果电脑上插了一个U盘，没有弹出直接硬拔了，会有什么影响？对什么造成了破坏？有多少同学没有硬拔过U盘？

操作系统跟磁盘会有cache。比如说你要往磁盘上写，磁头掉电有可能就把盘片弄坏了。现在应该不会有人大胆到把移动硬盘（HDD）硬拔——有可能就把硬件拔坏了。对于SSD或者U盘，能不能拔呢？拔了对物理硬件不会有多大损害，但损害的是什么呢？本质上，你有一个文件拷贝到USB上，拷贝完了，但有可能在缓存或者缓冲区里还没有完全flush到U盘上，这时候拔了，再插到其他电脑上，文件打不开、乱掉了或者corrupted了，因为cache没有清空。反过来从USB读到磁盘也是一样，有可能还有一部分没读过来，文件没有完全打开。所以平时让U盘弹出，就是为了把这些cache清空好。

没格式化过的同学回去找个U盘格一下，大概就有印象了。

一个磁盘格式化完之后，还得把它映射到一个目录上，操作系统才能用。一般分几种形式：IDE接口的磁盘在Linux里面映射的设备名字叫/dev/hda、/dev/hdb等等；SCSI接口的映射变成/dev/sda、/dev/sdb。first partition是第二个drive的sdb1。

一个磁盘最多支持四个primary partition（主分区），如果再多的就变成logical partition（逻辑分区）。现在正是折腾电脑的时候，可以去给它分几个分区。

磁盘有一个root partition，root partition上面装有操作系统内核和user space。root partition在boot的时候被mount，mount完了之后操作系统起来了，再慢慢把其他分区mount起来。mount过程中操作系统会检查是否所有的metadata是正确的，如果有问题会把它fix。boot block里面存的是boot loader等等。

磁盘里面还有一个概念叫MBR（Master Boot Record），里面有boot code，也有分区表。从MBR可以读到整个不同的分区。如果MBR坏了，这个磁盘就废了，这时候得拿光盘去recover MBR。

swap space management——交换区域，可以把它用单独的一个磁盘或者一个分区放上去。


## 九、磁盘的连接方式

磁盘是怎么连接到电脑上的？日常中见到的都叫host-attached，直接attached到bus上。也可以做network-attached，还可以做storage area network（SAN）。

看一下计算机体系结构的图，processor只占了这么小一点点，memory和cache也在这，底下有PCI总线，PCI总线连更慢的磁盘，还有IDE和其他bus，串口、extension bus连keyboard。

每次提到这个图我都会讲到一个比较老的概念——南桥和北桥。北桥是一个bridge，连的是高速的内存和CPU，速度比较高；南桥是一堆从北桥下来的，连接慢速的磁盘或者keyboard。一听北桥就知道速度快，一听南桥就知道速度慢。以后面试如果谁问你南桥北桥，你也得答出来点儿，但同时心里也得想想——这个面试官也是老古董了。

磁盘通过I/O bus attached到系统上，讲了两个协议：IDE和SCSI。在Linux上的程序里，要么是sd，要么是hd。还有一个叫Fiber Channel，是高速串行总线，用在SAN上。

NAS（Network-Attached Storage）——把磁盘加上一个很简单的client，连到网上，通过网络访问就像在本地一样。全称就是network-attached storage。

SAN（Storage Area Network）——如果NAS做的多了，可以加一个storage area network，是一个private network，用的是高速协议、高速互联，方便传大量数据。

磁盘变得越来越便宜，如果磁盘涨价，我们平常生成的垃圾数据就太多了，但总体来说磁盘的价格是慢慢下降的。


## 十、RAID结构

下面讲RAID——Redundant Array of Independent Disks。这里面有几个关键的点：第一个是redundant，第二个是independent。independent是说独立磁盘，一个磁盘坏掉了别影响其他的。redundant是这个技术的核心——通过冗余来提升reliability。

key insight是：disk不可靠，但是很便宜。怎么样通过不可靠但便宜的磁盘，用冗余来构成一个reliable的服务？这就是早期计算机科学家面临的问题。还有一个好处：如果把两个磁盘同时读，带宽就提升了两倍，增加了speed。

提到RAID就绕不开一篇论文——Google File System（GFS）。在我们包括之后好多年读书的时候，Google File System都是如雷贯耳的名字。它讲的就是怎么样用cheap但不reliable的磁盘来构建海量数据的存储系统，支撑当年Google的搜索。

RAID通过冗余来提升reliability和speed。关键技术有两个：第一个是data mirroring——mirror是镜子的意思，就是把data镜像一份，同样的data存两遍、三遍、四遍。第二个是data striping——把数据切成不同的块存在不同的地方。第三个是error correcting code（ECC）。

RAID结合了这几种技术，mix and match，看怎么样挑一种或多种，最后提升速率和reliability。

RAID分成好多level，从RAID 0到RAID 6，分成7个level。常用的有RAID 5+0、RAID 6+0。

RAID 0：把data拆分到不同磁盘上，没有任何冗余。好处是能提升速度——一个大文件拆开存到4个磁盘上，四个磁盘一起读，performance提升了4倍。但没有任何redundancy。

RAID 1：走了另一个极端，就是mirroring，把所有data存两份。优点是什么？可靠——一个磁盘挂了还能从另一个磁盘读。速率也提升了，可以从两个磁盘同时读。缺点是什么？费磁盘——本来能存1个T，这样一整只能存512G了。

RAID 2：把data在bit level做striping，用海明码做纠错——4个bit加3个parity bit，需要7个磁盘。RAID 2没有被用起来，原因是什么？关键点就在于bit level——从这个磁盘读，最少也得读一个byte，怎么可能在bit level做striping呢？存一个byte要存8个磁盘，效率特别差。因为切分太细了，所以RAID 2没有被用起来。

RAID 3：也没有把问题解决掉，也是bit level的striping，只加了一个parity bit。同样有bit level striping的问题。

RAID 4：和RAID 3很像，但是striping做的是block-based，不是一个bit一个bit地切，而是按block切分。block存在四个盘上，最后一个盘存parity block。

RAID 5：跟RAID 4一样，但是parity被分散在各个磁盘上。

RAID 6：跟RAID 5很像，只是除了P又多了一个Q，多了一位纠错码。

RAID 4、RAID 5、RAID 6可以单独用，RAID 2和RAID 3不能单独用，因为输在了bit level striping上——现在byte已经不是读出的基本单位了，读出来一般都是block，再去recover bit效率太低了。

RAID可以组合起来用，比如RAID 6+0或者RAID 5+0，结合起来再split一下。

RAID能protect against disk failure。问大家一个问题：现在要写一个磁盘，从内存往磁盘里拷，突然有一个bit flip了，bit翻转了，RAID能解决吗？不能。因为你交给磁盘去写什么，它就会写什么——本来bit是1它翻转成0了，磁盘的firmware以为你让它写0，就把0写出去，还给0配了纠错码，纠出来也是0。所以不能recover，这叫data corruption。RAID只能recover disk failure——磁盘挂了可以，但不能recover file system error或者data corruption。

那怎么办呢？文件系统比如Solaris ZFS加了额外的checksum来检测错误，ZFS给所有的FS data和metadata都加了checksum，可以检测和纠正data和metadata corruption。这个我们下周再讲。
