## 轮询与中断

然后我们CPU跟这个硬件、CPU跟device做交互的话，刚刚说两种方式。第一种呢叫polling，polling呢也被称为轮询。什么轮询呢？就是这个CPU他会一个设备一个设备的去问，问这个设备他有没有新消息要发给CPU。然后如果这个设备在忙的时候，他在处理一个操作，没有给他及时的回复，这个时候呢CPU呢就会busy waiting，他会不停的去读这个status register。如果这个status register显示的是busy的话，他会不停的去等、去读。他会陷入到一种busy waiting的状态。如果这个register现在显示的这个设备是空闲，他会去执行第二步，就是发这个command给这个设备。然后呢这个设备呢他自己做完这个操作，满足了这个执行的command之后呢，他会更新他的status register。这个时候呢CPU再去读他这个register。然后呢这个整个呢就算完了。

这个轮询呢它有一个不好的缺点，是他需要busy waiting。假如设备在忙，那CPU在等，一直这样等。那如果这个设备它很慢很慢，那你CPU要等的时间就会很长。所以他在这一点上，他的效率呢就会大幅度下降。

如果我们不想做轮询，我们不想让CPU去主动，因为CPU太快了，他做轮询呢会发生这个busy waiting。那我们怎么来做？早期的计算机科学家就想到，既然你设备慢，那你设备如果你执行了一个命令，执行完了之后，你能不能反过来通知CPU。在你通知之前，这个CPU他就在一直做CPU这块的事儿，比如说切换到其他的进程去执行。当你这个设备执行完之后，你来interrupt CPU，告诉CPU你ready了，你来处理。所以这个interrupt呢他就改变了，不是CPU主动，而是这个设备主动、CPU被动。他的一个好处呢就是这个CPU在这个设备忙的时候，CPU不用busy waiting，他不用忙等待，他可以去执行其他的进程。所以他这个效率呢整体来说是比较高。那他的缺点是什么呢？他的缺点是CPU执行其他的进程还要做context switch，他要做上下文切换。那这个里边就引入了一个问题，如果你上下文切换的时间甚至大于他的busy waiting时间，那其实你这个开销也是不值的。

所以这个interrupt和polling呢他们两个是被轮番使用。什么时候用interrupt，什么时候用busy waiting，他其实就有点像我们前面学的这个mutex lock和semaphore，跟这个其实很像的。那我们就提一个问题哈，什么时候用polling，什么时候用interrupt。从场景上讲一下。大部分情况用interrupt，好同学请坐哈。这位同学态度一直都很好，每次都抢答哈。下课告诉我你叫什么名字啊。然后下一位同学。好同学请坐哈，好下一位同学。你这个回答出来的是，确实是busy waiting比较短，context switch时间更少，然后呢在context switch时间更少就选用哪一个。但是他的根本的原因是什么？是这个设备它是快还是慢，还有一个呢就是你这个操作系统它的load到底是high还是low。如果这个设备慢，那你busy waiting的时间就比较长，会大于context switch的时间，这个时候呢设备慢，我们就选interrupt。如果设备比较快，busy waiting的时间一般很短，那他远远小于context switch的时间，那你就选择polling。

用interrupt他整个过程就变成这样了。就是呢这个设备呢会先initiate一个I/O，initiate I/O之后，就相当于它发一个request给这个I/O controller。大家注意这个I/O controller他一般是跑在设备上。然后呢在这个meantime，这个CPU呢他就去执行其他任务去了。但这个设备呢他很慢，他慢慢慢慢执行。执行完之后，他终于这个I/O执行完了，执行完之后他就发一个interrupt，发一个interrupt，CPU呢就要去handle这个interrupt，发现这个I/O任务已经满足了，他就把这个data读进来，然后这个任务呢就能响应了。

如果一个进程要读磁盘，操作系统呢就要去读磁盘，发一个request，磁盘慢慢去读，然后操作系统呢就回头做其他的事儿。然后呢等这个磁盘完成了之后，他会发一个interrupt，操作系统呢会把这个磁盘的这个数据ready，然后回到这个地方去，让那个读磁盘的进程继续执行。这就是整个的一个过程。然后interrupt机制呢现在几乎成了我们这个计算机系统的一个基石。

还有一个叫cache，cache呢它其实就是for fast access。这个我之前就跟大家讲过cache和buffer他们两个之间的差异。然后呢另外一个就是spool，spool呢你可以理解成它就是一个buffer，printing的buffer。还有一个就是device reservation，我们可以预留一个设备，用系统调用可以控制这个设备的lock和unlock。

另外一个呢，I/O也是最容易出错的地方。一般操作系统我们都要能从这个I/O里边给它recover出来。如果读和写没成功，我们要retry。一些操作系统呢有更好的recover机制，就比如说一个设备他一直发生错误，那我就把它给下线了。

还有一个就是I/O的protection。这个我们从计算机系统二里面就学了，操作系统要管理外设，他怎么样来管理外设呢？他又把所有的跟外设相关的寄存器都变成了privileged，只能在内核态运行。那你所有的user space想用它都得通过系统调用掉到内核里边。同样的，MMIO和I/O ports也都需要被保护好。

这就是一个典型的用户的系统调用。他如果想访问一个设备，比如他想屏幕上显示点东西、显示一个字符，或者从keyboard里面读一个字符，他都在trap到kernel里面然后再回来。这个大家现在应该已经比较熟悉了。

内核呢为了支撑这个，他就提供了一系列的data structure，有open file table、network connection table、device state。这个我们后续可以再讲。然后呢不同的操作系统呢，像Windows他还提供了把一个消息传到内核态，内核呢就不停的去处理这个消息，最后处理完之后再把它传到用户态。

这个里面就是我们这个打开的文件，比如说有文件，有一个open file table。这个open file table里面存有file descriptor record，还有不同的打开的文件，还有network connection table。这两个link我们到文件系统讲的时候会跟大家再详细展开讲。

## I/O请求的生命周期

系统调用我们怎么样来做，怎么样满足这个I/O request的。我们一般是先找到我们要读一个文件，先找到哪个设备存放的这个文件，然后把它转换成这个设备能理解、操作系统能理解的。然后读把这个文件读进buffer里边，然后让这个data可用。这是一个常用的图来展示他的整个生命周期。

我们从这个操作系统user space发了一个读文件的system call，然后我们的操作系统内核会问我们能不能满足这个请求。如果能满足，比如说文件已经被人读过了、在内存里，那就直接返回了。如果要不能满足，就很复杂，他会调用device driver，device driver还在内核里面跑，他会给device controller——这个就是在磁盘上跑的——发一个命令。然后在这个meantime，这个CPU就被调度去做其他事了。device controller他完成了之后，他会发一个interrupt。然后操作系统呢会接收这个interrupt，然后决定这个I/O已经完成了。最后再把这个数据返回给user space。这是整个的一个读文件的一个流程。

## I/O性能问题

然后这里面有一个performance的问题。什么问题呢？就是这个操作系统呢，他要做context switch。然后如果这里面还要做data buffering和copying，这里面的效率呢其实就降下来了。I/O呢它是影响操作系统的performance的一个最主要的major factor。这个我们还看不出来他这个问题很大，我们又找了一个更极端的一个例子。大家看一下这一个例子哈。

这个呢上面画的是硬件，这个硬件有keyboard，有network adapter，网卡。下边呢这个是user space的进程，中间呢是操作系统内核态。想做的这个事儿呢就是remote login。假设这是我本地的，这是远程的server，你们经常这样做，ssh到你们远程的那个server上。你比如说你在这输入一个命令，远端的server要接收到。那这个整个过程是什么样的呢？大家一起看一下啊。

最初是用户他要输入，比如说他的用户名和密码，他要按按键。我们说这是一个硬件。好，那他的按键呢要被操作系统来处理。他按了一下键，就会产生一个interrupt，操作系统收到这个interrupt呢，他就会处理这个interrupt。然后他要从这个device driver——keyboard device driver里边读到这个character，然后读到这个character在这中间呢会发生context switch，因为他有可能很慢。取回来之后呢，这个character拿到之后，操作系统呢就把它传给这个user space的程序了。

那怎么发呢？得通过network adapter。network adapter是一个硬件。所以操作系统呢又要介入，这个user space的程序呢它就要调用一个socket的发送。然后又要通过这个硬件，然后这个硬件呢他就要往这个网络里面真正的去发送这个包。发完包之后，他给操作系统回复一个“我已经发送了”。然后这边呢他的network adapter接收了之后，发出一个interrupt，操作系统来处理。拿到这个character，这边就是你要远程ssh那边他要做authenticate的那个进程，他拿到之后，他要做不停的context switch。所以你每输入一个字符，这个流程他都要走一遍。这个里面大家觉得他效率怎么样？效率低，好同学请坐哈。那下一个问题就是他效率如果比较低的话，怎么去优化？感觉这个时间花在哪儿啊？批量发送，同学都已经有优化的方案了。他这个效率低，就是因为这些都是硬件啊，他要不停的经过操作系统，back and forth。

那我直接让这个硬件跟这个用户态打交道。这俩硬件假设我都把它绑到这个用户态这个进程上，我是不是就可以把所有的操作系统这些操作都给它bypass掉？就相当于我把操作系统给他省掉，我让这个硬件直接talk to这个用户空间。现在有很多在云上，大家觉得这个东西特别没效率。然后呢提了一个概念，我们刚刚这位同学说我可以攒够一些字符一起发送。还有一个他们现在工业界做的更极端的，他叫I/O直通。就是这两个硬件它可以直通到这个用户态的这个进程上，中间这个内核就不在了。相当于你keyboard按了一下，我这个用户态直接收到，我直接通过网卡就发出去，中间就不经过内核了，这叫I/O直通。这个东西呢现在在云上、在server上已经变成一个很重要的方向了。就是把这个device直接映射到这个进程里边。

那他的缺点是什么？他的缺点是，如果这个进程出了问题，那这个系统就废了。他可以不停地bind这个keyboard，我就不让别人读；或者他可以一直占着这个network adapter；或者说别人敲keyboard的时候，这个进程也能读，所以他把别人的密码也读进来了。所以呢，大家为了优化效率，这些激进的优化效率的措施，也往往伴随着一系列的这些问题。

我们一起来看一个例子哈，就是Linux里面有一个设备叫TTY，这个就是输入输出设备，terminal。然后呢他有自己的initialization，有自己的初始化。大家可以看一下这个code。好，这就是他的初始化，这个东西只要知道就行，不需要懂。然后呢这个我绑定了一个设备，这个设备呢其实也是一个文件。然后我有tty的这个operation，就是它里面支持的operation。
