## 二、文件系统回顾：toy example

好，我们先把我上次课学的这个file system server的toy example——这个玩具的例子——跟大家讲完，再跟大家复习一遍哈。

我们上次说到，我们有64个block，每个block是4K，然后我们拿出来了56个给data block，作为存储数据的。所以它最高的磁盘使用率只有56除以64，等于87.5%。它最高只有这么高了，因为其他的存了一些辅助的信息。

### 2.1 辅助信息的存储

辅助信息存哪些呢？首先我们得有一个inode table，我们拿5个block出来分配为inode。然后每一个inode是256个byte，一个4K的block能hold 16个inode。好，剩下的呢，我们就存data block。Inode的管理我们可以用bitmap。所以我们这个存储就比较富裕：一个存inode bitmap叫I，一个存data block bitmap叫D，这是两个block。最后一个呢是superblock。Superblock存什么呢？它存：bitmap从哪开始，inode的范围是从哪到哪，data block从哪开始。

好，有了这样的一个toy存储之后，我们就可以实现——拿到inode number，我们直接就能计算inode的位置。比如说给大家一个例子：inode 32，如果我们想算它的位置，我们就算——32，一个block存16个inode，所以32除以16，它就是在第二个block的开头。所以在这里面，一个block是4K，所以它的开头就是8K。那这个inode block从哪开始呢？从12开始，再加8，所以就是20K。

好，再给大家快速算一个——选个吉利的数，19。它的位置从哪开始的？假设给你一个inode 19，你怎么算？能不能套这个公式？很简单，你就用19乘以sizeof(inode)的大小，再加上它的起始位置。起始是在12K的那个地方，12KB。好，这就是inode 19所在的位置。期末考试的时候要跟大家出这个题，你得会哈，要不然这个太简单了——这个是inode怎么样算它的位置。

拿到inode之后，它里面就存了data block的指针。然后它有12个direct直接指向data block，还有一个single indirect、一个double indirect、一个triple indirect。这个很简单，会算一下最大的file size是多大。

### 2.2 Directory与文件读取流程

后面的一个关键点就是directory——他是一个文件，他的内容存的是directory entry，这个要记好。我们又到最后的这个综合example了。假如我们要去读/foo/bar这个文件，操作系统首先呢，根据我们前面的例子，他会去读root的inode。我们把这每一个都挨个过一下哈——从上往下是时间顺序，越往下越靠后，越往上越靠前。

第一个read，他要读root的inode。你可以回答一下：他是读谁的inode，读什么内容？——这个里面两个关键点：第一，root inode是谁的inode？他就是"/"（slash）的inode。读它的inode是为了干什么呢？是为了读directory entry。Directory entry是在data block里存着的，inode里存有data block的号，所以他要去读data block的指针。第二个read呢，读data block里面的内容，这个内容是directory entry。读完directory entry，他就去把整个读来扫一遍，想找foo的inode是多少，然后就找到了foo的inode。

所以下一步去读foo的inode——因为foo也是个目录，所以他实际上要去读foo的data block的指针，然后去读它的data block。读到了bar的directory entry之后，下一步就找到bar的inode，然后知道bar的data block指针是哪些。所以下面这几个就是读它相应的data block。还有一个write是写时间戳——我们上次课讲过，这个设计效率特别低，所以后来Linux用了一系列优化把它给优化掉了。

### 2.3 文件写入流程

写这个就比较复杂了。写的时候呢，它还是先找一下到底bar存在不存在。先读root的inode，然后拿到它里面的所有directory entry，找到foo的directory entry，找到foo的inode之后，然后去读foo的inode，把foo里面的directory entry读完，发现bar不在。这就要分配——说明文件不存在，我要创建一个，分配一个inode。所以读bitmap、写bitmap，把inode bitmap给占了。

分配完之后，我们有inode，也有文件名了。这个write是写什么的？——他是写foo的directory entry，把bar这个文件名和bar的inode作为一项directory entry，放在foo的data block里面。所以这个就是写。然后下面时间戳也写一下。接着就是bar里面写内容，因为它的data block不存在，所以得去读写data的bitmap。

好，这个例子就跟大家又回顾了一遍。我觉得这个例子还是挺好的，跟我们前面配合得天衣无缝。如果期末考试要出一个这样的题目，你们应该会写。

## 三、系统安全概述

好，我们这个文件系统就讲这么多。我们上次课其实已经把security开了一个头了。如果有人问你这个计算机或者计算机系统到底安全不安全，你说其实是有标准的——你可以讲到TCSEC、ITSEC、CC，还有国内的国标。

另外一个呢，你要知道这个东西是trust的，是trust什么，然后它的attack surface是什么。TCB就是trusted computing base，attack surface就是所有能攻击到它的路径入口。还有defense in-depth。我回去回顾了一下，才发现common concepts里面少讲了一个东西——thread model。这两个词谁知道什么意思？——翻译成中文一般叫威胁模型。啥叫威胁模型呢？就是当你设计一个防护或者攻击的时候，你要先讲清楚：攻击者有什么样的能力，防守方有什么样的能力，然后你的TCB是什么东西，这就叫threat model。所以threat model概括两个东西：第一个是attacker的capability；另外一个你得定义你的TCB到底是什么——你相信什么东西肯定不会出错。

以后你们如果要搞一个安全的东西，讲一个攻击或做一个防护，别人说“我可以这样攻击你，也可以那样攻击你”，你跟他讲不清，就是因为你的threat model没定义好。你先说我能防什么样的攻击，在这个场景下attacker的能力是什么，然后我这个防护能把这个攻击都防住。好，这就是threat model很重要的一个概念。

讲完threat model之后，我们讲了authentication和authorization这两块。今天就把后面给大家讲完。

## 四、安全评估标准

我们还是从common concepts开始，一点一点把都给大家回顾一遍。

我们讲了这个security evaluation criteria有这几个：TCSEC、ITSEC、CC、GB17859。TCSEC比较老了，它叫Rainbow Book，七八十年代DOD就已经把它发布出来了。这个里面有一个常识：你的电脑，不管是Windows、Linux还是Mac，大部分处于C级。D是最小保护，基本上没有protection；C是Discretionary Access Control，自主访问控制；B是Mandatory Access Control，强制访问控制；A是Verified Protection。

C这个单词是"自主安全"的意思。一般你这个电脑的owner可以自己去改安全属性，这就是DAC。像Mandatory Access Control，比如说一个保密机上面有这么多规则，你根本改不了，它是属于B级的。你自己能改的都属于C级的，所以你们现在用的电脑大部分都是C level。

再往后呢，欧盟出了一个ITSEC。再往后就是CC——Common Criteria。不知道大家注意过没有，不管是小米手机发布还是华为鸿蒙手机发布的时候，都说通过了一个什么认证——一般都是通过了CC的认证，相当于EAL5+或EAL6。这个CC就是西方一起搞的。然后相对应的，国内搞的就是国标GB17859，这是对应的关系，你们自己再去看一下。

## 六、访问控制：Authentication、Authorization、Auditing

Protection这一块，Authentication、Authorization还有Auditing，都是Access Control的三个基本的实践。Authentication是认证，Authorization是授权，Auditing是审计。

### 6.1 Linux SUID机制

我们讲了Linux的SUID。你们如果看Linux，比如十年前的Linux，你会发现ping是一个SUID的程序——为什么ping是SUID的呢？因为ping需要发送原始套接字报文，普通用户没有这个权限。后来ping从SUID改成了基于capability的了。SUID就是给他root权限，他是个超人，我们讲了它不安全，然后引入了capability。Capability把root的能力切成40多份，但是切得不均匀，其中三分之一的能力都给了CAP_SYS_ADMIN。上次课就讲到这儿。

这套机制我们曾经花了很多时间想全面优化一下，但是因为kernel说不能break以前的老程序，改了之后还得能运行，这个要求达不到，就不了了之了。

### 6.2 ACL与Capability

ACL和Capability各有哪些优缺点？现在的系统里面，一般用ACL去open一个object的时候，围绕object来管；去做一个操作的时候用capability。Linux里边这两个是同时都有的。

### 6.3 Auditing

Auditing审计要求执行的这些操作都有记录、有log。Log有什么用？方便后续去做分析——一查就知道事件是怎么发生的，给了谁什么权限，访问了哪些资源。事后出了问题可以做分析。

Auditing比较重要，但大家觉得懒，都懒得实现。CC标准有一个硬性要求，一定得有audit framework。Linux为了满足CC，提供了一个audit subsystem，在kernel/audit里面一大堆东西。现在你们用的Linux其实很少有auditing，但到工业界去构建infrastructure的时候，auditing很多时候还是很有必要的。

### 6.4 Reference Monitor

基于Authentication、Authorization和Auditing，我们实现了一个东西叫Reference Monitor——引用监视器。它是什么样的概念呢？假设你有一个东西锁到寝室里了，所有同学想拿这个东西，都得通过寝室大门，寝室大门就像一个reference monitor。想读你那本书，都得经过寝室大门。这就是reference monitor——有一个保护的对象，就是那本书；主体就是你的同学；你在大门口看着，有谁进来，你可以给他授权。先看看他是谁，认识的才授权——这里面有authentication和authorization。

Reference monitor如果要正确实现，有几个要求：必须不能绕过。假设你这本书放在这，你觉得很保险，但同学发现有个窗户，不从大门进，爬窗进来把书拿走了，那reference monitor一点用都没有。所以要求reference monitor不能被bypass，要考虑所有访问路径并保证所有路径都能通过检查才行。

举一个例子：应用通过文件描述符fd来访问文件，没有办法直接访问磁盘或通过inode访问文件数据，这个时候文件系统就是一个reference monitor。

### 6.5 Reference Monitor的难点

Reference monitor在我们计算机里很常见。Rust语言里，分配一个object有自己的ownership，如果把reference赋值了，老的reference就失效——只有一个reference指向它。这实现了一种ownership，继承自reference monitor这个概念。

大家可能觉得non-bypass很容易实现，其实系统大了特别难。举个例子，这是一个敏感操作，怎么样保护？我在函数开头放一个reference monitor，那是不是就可以了？不行，因为这个函数A里面调用了B和C，攻击者不调A，直接调B和C就把reference monitor给绕过去了。那把B和C里面也各放一个行不行？还是不行，因为B和C也调了其他函数。这样嵌套下去没完没了——这也是为什么在大型系统里reference monitor特别难做。

C语言里面这个东西很容易出错。假设你分配了一个struct，想访问它怎么访问？取地址呗。地址是个数，我只要拿到这个数，就可以把一个数转化成对这个struct的reference。C语言太灵活了，导致好多问题。地址一泄露就打出来了，直接强转成指针就访问了。在我们搞系统的人眼里，哪有什么类型，就一个地址和地址里存的东西。所以C语言特别不安全，Rust才出来的。Rust说你别用这个数来引用我，给你规定一个reference类型，并且有ownership——你一旦把这个reference给了别人，原来的就没了。Rust很好地实现了reference monitor这个思想。

Reference monitor要注意的就是non-bypassable，不要很容易就被bypass。

## 七、DAC与MAC

DAC——Discretionary Access Control，自主访问控制。指对象的拥有者有权决定这个对象是否可以被其他人访问。文件系统就是典型DAC，文件拥有者可以设置权限。但部分场景如军队，DAC过于灵活——文件拥有者是否真的有权随意设置文件权限？

所以有了MAC——Mandatory Access Control，强制访问控制。由系统增加强制的、不可改变的规则。比如军队里某个文件设为机密，指挥官也不能给没权限的人看——这个规则是系统规定的。DAC和MAC可以结合使用，但MAC优先级更高。

MAC后来被引入Linux，就是SELinux——Security Enhanced Linux。在SELinux里有一套policy来控制哪个subject能访问哪个object，由系统管理员load，就算有root权限也不能重新load这个policy。SELinux后面衍生出了SE Android。现在所有Android都实现了SE Android，有一套policy，手机owner改不了。每个app自动被归为untrusted domain，能访问的资源受限。你们天天在用，只是不知道而已。

## 八、Protection与Security的区别

很多同学问Protection和Security到底有什么区别？Protection主要是讲Access Control——你能去做访问，规定谁能访问什么。Security的范围是什么呢？系统里有bug，access control被人突破了怎么办？这就是Security的范畴。

## 九、三种攻击类型

常见的安全攻击总结有三种形式：第一种是Code Injection Attack代码注入攻击，第二种是Code Reuse Attack代码重用攻击，第三种是Non-control-data Attack非控制数据攻击。

来看一段代码——假设有一个程序，里面有一个bool值，如果是true就给root access，否则给normal user access。这是控制流图，有一个判断。现在假设有一个bug，攻击者可以做三种攻击。

第一种：bug太强，系统没防护，可以往系统里注入代码。直接注入一个call root access，就拿到了root权限——代码注入攻击最强。第二种：如果不能注入代码，只能改控制流——code reuse攻击，让他直接跳到root access那里。第三种：不能改控制流，也不能注入代码，那就把bool变量改成true——永远拿到root access。

这三个分类和业界分类不谋而合：能执行新的代码、能执行老的代码但以非预期顺序执行、代码以预期顺序执行但数据变了——即non-control-data attack。

## 十、攻防演化

攻防一直没有终点，到现在再安全的系统，只要花足够时间也能被攻破。如果做到百分百安全，我们岂不是都下岗了？攻防一直在对抗中演化——魔高一尺道高一丈。攻击的演化复杂性指数级增加。二十年前能写代码重用攻击是大神，现在课程要求写内核代码重用攻击。代码注入攻击很简单，代码重用攻击难得多，数据攻击又更难——隐蔽性也在增加。

但好的一点是攻击者的控制能力在缩小。就算最小的数据攻击，依然能root内核。

### 10.1 代码注入攻击时间线

最早1988年Morris Worm，MIT的学生Morris写的蠕虫，感染了当时全世界为数不多的Unix机器。后来这人被罚了社区服务，现在成了MIT教授，因祸得福了。Morris Worm的原理就是把代码直接注入系统执行。

真正防住代码注入攻击是2001年——硬件架构有了改动，页表里加了NX bit（不可执行）。AMD率先实现，Intel随后跟进。最早应用的是Windows操作系统，把代码注入攻击几乎彻底防住了。现在你的电脑上几乎不可能做内核代码注入攻击。

### 10.2 代码重用攻击

代码注入被防护后，攻击者想：不能注入新代码，能不能重用已有代码？最早1997年黑客Solar Designer提出return-to-libc攻击，不注入新代码，返回到libc函数实现攻击。2014年微软Windows出了CFG（Control Flow Guard）防护；同时三星做了CFI；ARM出了硬件机制PA；Intel出了CET（Control-flow Enforcement Technology）。这才算把代码重用攻击完全防住。

代码重用攻击的基本思想就像电影里绑架信——从报纸上剪字拼成一句话。从现有程序里选代码片段（gadget）拼成想要的函数。两种拼法：ROP——通过改返回地址；JOP——通过改函数指针。防护方法：保护返回地址用Stack Canary（金丝雀）；保护函数指针用跳转检查；还有ASLR和KASLR地址随机化。

### 10.3 非控制数据攻击

代码重用被防住后，攻击者又开辟新战场——非控制数据攻击。2017年有真实的内核攻击。Linux内核4.17版本有一段代码——ss_initialized是个全局变量，放在data段，所有进程都能访问。如果攻击者把它写成0，security_compute_av函数里检查ss_initialized就始终为false，直接goto allow，整个SELinux就被bypass了。国内大神给这起名叫"点穴"——几十万行代码的SELinux，点一下这个穴位就全废了。

非控制数据攻击至今没有统一的保护方案，主流操作系统缺乏有效防护。Apple做了一些前沿保护，把关键数据用ARM硬件机制PA签名保护，但大部分Android和Linux还是没有保护的。

## 十一、攻防总结与内存攻击分类

我们组的包括以后课程会往智能架构方向去改——AI框架、推理框架、GPU/NPU。AI模型安全研究集中在数据投毒和后门，忽略了软件安全。推理框架如TVM、MLIR、SG-Lang直接开端口，任意地方能访问。GPU/NPU的攻防都还没怎么研究。我们组在这方面做了很多工作，发了安全顶会论文，还拿了今年的杰出论文奖——中国唯一。

内存攻击的分解：指针只能out-of-bounds读或写。读造成信息泄露，写是code injection attack，写控制流是control-flow attack，写data就是non-control-data attack。
