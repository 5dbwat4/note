# 知识点

SQL——基本SQL和扩展SQL。这部分比较适合出填空题和判断题，SQL语句嘛。所以主要考察大家像create table这样的语句，这里要注意数据类型——数字型还是什么，数据的类型；还有一直在强调的约束条件、完备性约束；还有key，比如主码、超码、候选码、外码，很关键，包括foreign key可能会有一些级联操作，像cascade等；还有授权、权限设置。这些知识点大家要注意。对于SQL语句，比较容易出判断和填空。判断题经常尤其涉及聚合函数，大家一定要注意，这里很容易出判断题。聚合函数包括count、max、min、average、sum，就是求平均、求和、求最大、求最小、求个数。这些里面有一些注意点。还有，比如假设给你一个查询请求以及一个写好的SQL语句，要你判断SQL语句中某部分写得对不对——相当于需求告诉你，SQL语句也写好，但里面有正确有错误。这些SQL语句包括from、where、group by（涉及聚合函数的分组）、order by排序等，你要判断这些关键字怎么用，要注意什么，实现什么功能，要学会判断。所以难度已经降了不少，不要求你写，只要求你看得懂，根据需求判断哪个对哪个错，包括聚合函数的也要会判断。OK，这就是SQL语句主要知识点。

第二，ER模型。ER模型相对比较容易出选择题。ER模型里要了解实体与实体之间的关系——一对一、一对多、多对一、多对多。还有各种属性类型，尤其要注意派生属性,导出的属性和多值属性。这些属性类型需要注意。还有弱实体，对应的强实体。实体之间有关系，还有全参与和部分参与——全参与就是整个实体集合中的所有值都包含在关系里；部分参与则只有部分。比如把我们班所有同学都归到某一项任务，就是全参与；如果只有部分同学，就是部分参与。还有ER模型到关系模式的转换，有些关系不需要转成一张表，比如弱实体可以合并到它所对应的宿主实体那里去。这种题目，比如给定一个例子，要你选择属于哪种关系或实体，这些知识点不难，掌握了选择题很好选。

第三，关系规范化。给定一个关系模式（relational schema），再给一组函数依赖。要你列出候选键(candidate key)，这可以出选择题。还有正则覆盖（canonical cover）——左侧右侧能去掉的冗余属性和冗余函数依赖都去掉，剩下最小的覆盖，这个要掌握。还有分解，三大特性：第一，无损连接，不能分了之后再合并少了或多了一个；要判断怎样符合无损连接；第二，依赖保持，分解后函数依赖依然存在，不能少或多；第三，BCNF的分解——如果分解成只有两个属性，一定符合BCNF。这些知识点都要注意，比较容易出选择题。

第四，第九章存储。存储里概念性东西较多，也容易出选择题。比如那张类似光盘的图，磁头、寻道时间，从磁盘读取数据块的时间——磁盘数据读取时间主要包含寻道时间和读取时间，这个还记得吗？还有廉价磁盘阵列RAID，0、1、2、3、4、5等不同级别，每个级别特点、适用范围，不需要大家记，看明白就行。比如RAID1适合哪种场景，有什么特点，你可以在纸上记。还有缓冲区管理，命中率、缺失、替换策略等要掌握。还有不同文件组织方式的特点和适用场景，比如堆文件、顺序文件、散列文件、聚集文件等，它们的样子和使用场景要了解。还有定长和不定长记录，以及自由列表如何维护管理，也要了解。

第五，B+树。B+树最重要的知识点是，这种题目肯定容易出选择题：告诉你一棵B+树，插入或删除某个记录后变成什么样，下面画几个图让你选正确的；或者给定B+树和数量值，让你求树高，有公式；或者给定查询计算访问数据块的最大数量等代价估计；还有LSM树，要了解它一般在什么情况下使用，以及它的结构。索引主要目的是加速查询。还可能出一些创新题，比如大模型时代的learning index，描述一下他的基本机制，然后说他失败了，然后对比B+树会是什么情况，这个相对灵活。

第六，查询处理。也容易出选择题。查询处理的一般步骤，那张图：用户输入查询请求，先编译，再生成表达式，基于统计做选择、估计。还有排序（外部归并排序）、不同的连接算法（块嵌套循环连接、索引嵌套循环连接等）。还有评估表达式的物化和流水线，有什么优势特点，也了解一下。

第七，查询优化。比较适合出判断和填空。查询优化首先要等价——满足相同需求的不同表达式是等价的，要会判断两者是否等价；然后估算代价，比如选择率，以及元组数量、代价估算等，这些都要会。

第八，事务与并发控制。可以出选择题。要了解优先图，判断是否有环、死锁，死锁的判断方法，死锁发生后如何解决，如何预防死锁。还有饥饿状态——一个事务始终得不到满足，可以用超时来避免。两阶段锁协议也很关键，要学会判断事务是否可恢复、是否无级联、是否冲突可串行化、视图可串行化等。

最后，恢复。数据库管理系统失败的原因——软硬件、事务、系统、磁盘等；如何避免，比如基于日志的恢复。基于日志恢复中提到一个Read Ahead，要先写日志再做各种操作。有两种方式：延迟修改和及时修改。还有检查点，引入检查点有什么用，如何判断哪些事务需要undo、哪些需要redo。还有Aries算法，基本思想和基于日志的恢复一样，只是用了高级数据结构。要了解Aries算法处理流程，了解CLR（补偿性日志记录）等结构类型和用法。
